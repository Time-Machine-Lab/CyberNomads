## Context

任务拆分 Skill 现在承担的不只是“把引流策略拆成若干任务”，还要在拆分过程中判断需要哪些平台 Skill、哪些共享知识、哪些任务数据文件，以及什么时候把最终任务集安全写入系统。这里面至少有两类动作不应该交给 Agent 直接自由操作：

- 资源复制
- 任务批量入库

前者需要跨平台路径兼容与目录边界控制，后者需要事务性和明确错误反馈。因此这部分更适合被定义为 Cybernomads 提供给 Agent 的受控工具。

## Goals / Non-Goals

**Goals:**

- 定义一组面向任务拆分阶段的最小必要受控工具。
- 让资源复制行为保持在允许的运行时目录边界内。
- 让批量任务保存适合作为 Agent 工具入口，并具备明确失败原因。
- 与现有任务模块保持职责一致，不让 Agent 绕过任务模块直接入库。

**Non-Goals:**

- 不在本 change 中重建任务拆分 Skill 本体。
- 不在本 change 中设计任务执行阶段的全部工具。
- 不在本 change 中开放任意文件写入能力。

## Decisions

### 1. 提供两个最小必要工具：资源拷贝工具与批量 task 保存工具

这次先只把用户明确需要、且边界比较清晰的两个工具正式化：

- `copy_runtime_agent_resource`
- `batch_save_tasks`

这样能先支撑任务拆分闭环，又不会一次把文件系统开放面做得太大。

### 2. 资源拷贝工具只允许在受控目录之间工作

资源拷贝工具的源目录限定在运行时 `cybernomads/agent/` 下，目标目录限定在 `cybernomads/work/<trafficWorkId>/` 对应的上下文子目录下。这样可以避免 Agent 借“资源拷贝”名义访问任意本地路径。

### 3. 批量 task 保存工具沿用任务模块的受控写入语义

任务模块已经拥有“按引流工作创建或替换任务集”的领域职责，因此批量 task 保存工具不应另起一套存储逻辑，而应复用任务模块的受控入口。真正新增的是：

- 工具调用语义
- 结果格式
- 错误原因清晰度

### 4. 错误返回要面向 Agent 可消费，而不只是面向开发者

如果保存失败，不能只返回一个模糊的 500 或“validation failed”。工具结果至少要让 Agent 知道：

- 哪个任务失败了
- 失败在哪个字段或哪个规则
- 是否整批都未保存

这样 Skill 才能基于失败原因继续自检或中止。

## Risks / Trade-offs

- [Risk] 如果资源拷贝工具开放过大，Agent 可能越界复制任意本地文件。  
  → Mitigation: 严格限制源目录与目标目录范围，并做路径规范化校验。

- [Risk] 批量任务保存如果不是事务性，会导致部分任务成功、部分失败，破坏任务集一致性。  
  → Mitigation: 保持整批写入的一致性语义，要么整批成功，要么整批失败。

- [Risk] 只提供两个工具后，任务拆分 Skill 后续可能还需要更多上下文写入入口。  
  → Mitigation: 本次先最小化落地，后续如有必要再单独提案增加受控文件写入工具。

## Migration Plan

1. 定义新的任务拆分支撑工具 capability。
2. 更新任务模块 runtime 规格，使其适合作为批量 task 保存工具后端。
3. 后续实现时，将这些工具挂接到 Agent 可调用入口。
