## Context

引流工作域已经有完整的领域设计文档，但仓库中目前还没有对应的顶层 API 契约、SQL 契约和主 OpenSpec capability。现状是：
- `docs/api/` 只有 `products.yaml` 与 `agent-services.yaml`
- `docs/sql/` 只有 `products.sql` 与 `agent-services.sql`
- `src/modules/traffic-works/` 仍为空

这意味着如果直接进入代码实现，团队会在“对象绑定怎么表达、状态怎么拆、详情该返回什么、删除/归档/结束的边界是什么”这些问题上边做边猜。先补契约可以把实现边界固定下来。

## Goals / Non-Goals

**Goals:**
- 为引流工作域建立独立的 API 契约、SQL 契约和主规格。
- 保持“产品引用 + 策略引用 + 对象绑定 + 生命周期状态 + 上下文准备状态”作为本域最小核心。
- 明确契约与任务域、调度域、日志域、平台脚本域的边界。

**Non-Goals:**
- 不在本 change 中编写后端业务代码。
- 不定义任务表结构、任务日志表结构或调度算法字段。
- 不把对象绑定展开为独立资源域数据结构细节。

## Decisions

### 1. 引流工作契约只保存引用与状态，不保存产品/策略全文

SQL 与 API 契约都围绕 `productId`、`strategyId` 和对象绑定结果建模，而不复制产品/策略 Markdown 正文。这和领域文档中的“引用存储约束”“快照执行约束”一致，也避免工作域侵入产品域与策略域。

### 2. 生命周期状态与上下文准备状态分离

契约层显式区分：
- 工作主状态：`ready` / `running` / `ended` / `archived` / `deleted`
- 上下文准备状态：`pending` / `prepared` / `failed`

这样可以准确表达“创建失败但主状态仍停留 ready”“暂停后回到 ready 但不代表重新准备”的业务语义。

### 3. 对象绑定在契约层保持聚合表达，不提前拆成独立关系模型

由于对象域本身尚未完整落地，本次契约只要求引流工作能表达“某个策略对象槽位绑定到了什么资源对象”，而不强制在 SQL 层就把它拆成多表模型。这样能避免现在的契约反过来绑死未来对象域实现。

可选替代方案是直接设计 `traffic_work_object_bindings` 关系表，但当前对象类型、引用范围和跨域约束仍是高变内容，过早拆表会让契约比领域还重。

### 4. 列表摘要与详情视图都属于引流工作契约

本域要服务前端“工作区视图”，因此契约层需要同时定义：
- 列表摘要
- 详情视图
- 绑定摘要
- 当前上下文准备状态

这不等于把前端页面设计混入领域，而是明确本域需要对外稳定提供哪些读模型语义。

## Risks / Trade-offs

- [风险] 对象绑定先使用聚合表达，未来对象域落地时可能需要补充更细的关系结构。
  → 先把 API/SQL 契约设计为可演进表达，避免把对象域实现细节写死在第一版。
- [风险] 当前不定义任务表结构，后续任务域实现时需要再次新增 SQL/API 契约。
  → 这是有意为之，保持工作域与任务域分离。
- [风险] 删除状态既作为业务语义又可能和物理删除实现冲突。
  → 契约中只定义业务语义，不规定底层物理删除策略。

## Migration Plan

本 change 只新增顶层文档与主规格，不涉及代码迁移和数据迁移。后续运行时实现 change 必须以这里产出的 API / SQL 契约为准。

## Open Questions

- 对象绑定在 SQL 契约中是否长期维持聚合字段，还是在对象域稳定后再拆成关系表；本次先按聚合表达处理。
