## Context

当前账号池前端位于 `cybernomads-ui/src/pages/accounts/`，页面已经有视觉初稿，但数据来源和交互语义仍然是 mock 版本。`cybernomads-ui/src/entities/account/api/account-service.ts` 在 `useMockApi=false` 时会直接抛错，且当前 `AccountRecord` 仅包含 `id/name/owner/uid/status` 这类演示字段，与真实后端账号契约中的 `accountId/displayName/remark/platformAccountUid/lifecycleStatus/authorizationStatus/availabilityStatus` 并不一致。

后端账号模块已经完成一期运行时实现，真实接口可覆盖：

- 账号列表、详情、创建、更新、逻辑删除、恢复
- 授权尝试启动、授权验证、可用性检查
- 脱敏详情与授权/可用性状态读取

这意味着当前的主要问题不是“后端能力缺失”，而是前端仍在使用不兼容的页面模型和交互假设。与此同时，前端其他模块仍然是 mock-only，因此不能通过全局关闭 `VITE_USE_MOCK_API` 的方式来切换账号模块，否则会影响资产、策略、工作区等其他页面。

约束条件：

- 本轮不新增 `docs/api/accounts.yaml` 或 `docs/sql/accounts.sql` 顶层契约。
- 本轮前端必须复用现有账号模块 HTTP 契约，不能要求后端额外返回二维码图片、敏感凭证原文或历史记录。
- 当前账号页属于初稿，允许较大幅度调整页面内容与交互布局，但应尽量保留已经成型的视觉骨架和路由入口。

## Goals / Non-Goals

**Goals:**

- 将账号列表页和账号详情页切换到真实后端账号接口。
- 在前端建立与后端账号契约对齐的数据层，至少拆分为 API DTO 与页面展示模型两层。
- 让账号页面的一期交互与后端真实语义保持一致，包括资料保存、删除/恢复、授权尝试、授权验证和可用性检查。
- 为本地联调补齐账号模块专属的真实 API 路由方案，同时保持其他模块继续使用 mock。
- 明确当前页面中哪些区域可以真实接通，哪些区域需要占位或禁用处理。

**Non-Goals:**

- 不实现新的账号后端接口，不扩展二维码 challenge、凭证历史或终端日志 API。
- 不在本轮将所有前端模块统一切到真实后端。
- 不追求账号页最终视觉稿，只完成与真实后端契约一致的一期可用页面。
- 不实现真实平台二维码脚本或新的凭证消费接口。

## Decisions

### 1. 账号模块采用“模块级真实 API 接入”，不走全局 mock 开关切换

**Decision**

- 账号模块前端将独立接入真实后端，不依赖全局 `VITE_USE_MOCK_API=false`。
- 账号模块自己的 API 封装改为真实请求；其他尚未改造的模块继续保留 mock-only 行为。

**Rationale**

- 当前其他模块仍然直接依赖 mock runtime，全局关闭 mock 会让非账号页面整体不可用。
- 账号模块已经具备完整后端能力，适合先作为首个“单模块真实接入”的前端样板。

**Alternatives Considered**

- 直接全局关闭 mock 开关
  - 放弃原因：会同时打断资产、策略、工作区等页面。

### 2. 前端账号数据层拆分为“后端 DTO + 页面展示模型”

**Decision**

- 前端账号模块新增与后端契约一一对应的 DTO 类型。
- 在实体层或页面层引入 mapper，将 `AccountSummary` / `AccountDetail` 映射为前端列表行和详情页所需的展示字段。
- 页面不直接消费后端原始 DTO，也不继续沿用旧的 `AccountRecord` 演示模型。

**Rationale**

- 后端契约表达的是业务真值；页面需要的是 icon、label、tone、chip 文案等派生展示语义。
- 如果不拆层，前端后续每个组件都会自己拼接状态映射，导致状态语义分散。

**Alternatives Considered**

- 直接把后端字段名改造成旧 `AccountRecord`
  - 放弃原因：会继续保留 `owner/status/uid` 这类与真实契约不一致的历史包袱。

### 3. 页面交互按真实后端语义重写，不再保留“直接改状态”的旧动作

**Decision**

- 列表页：
  - 使用真实账号摘要展示平台、身份、标签、状态和更新时间。
  - 平台汇总卡由列表结果前端聚合，不新增统计接口。
- 详情页：
  - 平台和 UID 作为稳定身份只读展示。
  - 可编辑区域只保留显示名、备注、标签和平台扩展字段。
  - “保存凭证配置”改写为授权尝试 + 验证授权流程。
  - “验证连接”改写为可用性检查动作。
  - 删除/恢复接真实接口。

**Rationale**

- 当前详情页的 `updateAccountStatus(connected)` 是 mock 语义，后端不存在对应接口。
- 账号领域已经定义平台和平台 UID 不能在管理页直接修改。

**Alternatives Considered**

- 保留现有按钮名称和状态语义，只在内部偷偷映射
  - 放弃原因：会让页面动作名和真实行为长期错位，后续更难维护。

### 4. 不受契约支持的页面区域本轮降级，而不是反向驱动后端扩接口

**Decision**

- 下面这些区域在本轮不做真实接通：
  - 二维码图片展示与刷新
  - 敏感凭证原文回显
  - 凭证历史记录
  - 终端日志面板
- 页面上对这些区域采用占位说明、禁用态、只读说明或临时隐藏策略。

**Rationale**

- 后端当前刻意只公开脱敏详情，不返回敏感 payload。
- 前端一期目标是“先打通真实账号管理流程”，不是提前逼出下一轮契约。

**Alternatives Considered**

- 为了保持现有 UI 完整度，先临时扩几个后端接口
  - 放弃原因：会绕开既定领域边界，且真实二维码流程尚未稳定。

### 5. 本地联调优先使用 Vite `/api` 代理，而不是先改后端 CORS

**Decision**

- 前端开发环境通过 Vite 代理将 `/api` 请求转发到账号后端服务。
- 后端本轮不因为前端联调而优先引入跨域支持或浏览器侧额外协议适配。

**Rationale**

- 前端已经默认以 `/api` 作为 base URL，代理是最小变更方案。
- 当前后端是纯 Node HTTP server，优先保持后端实现稳定，把开发环境对接问题收敛在前端工程侧更稳妥。

**Alternatives Considered**

- 先在后端补全 CORS / OPTIONS 处理
  - 放弃原因：这会扩散到所有模块和部署面，不是账号页一期联调的最小闭环。

## Risks / Trade-offs

- [前端账号页改造幅度较大，可能和现有初稿视觉结构发生冲突] → 优先保留页面骨架与信息分区，只重写数据层和动作层，必要时再调整布局。
- [旧页面字段和后端真实字段差异大，映射规则可能散落] → 把状态映射和文案派生收敛到统一 mapper 层，不允许组件内重复拼接。
- [账号模块单独真实接入后，前端会同时存在 mock 模块和真实模块] → 明确账号模块使用独立 API 实现，避免继续依赖全局 mock 开关。
- [二维码区、凭证区降级后，产品观感会暂时弱于视觉稿] → 在页面上明确显示“待后续平台授权能力接入”之类的占位信息，减少误解。
- [列表统计卡由前端聚合，后续若数据量变大可能存在性能或一致性限制] → 本轮接受前端聚合；若未来需要跨页统计或大量数据，再单独引入汇总接口。

## Migration Plan

1. 在前端账号实体层新增真实后端 DTO 类型、API 调用和展示 mapper。
2. 改造账号列表页，先完成真实数据渲染、状态映射和行级动作接线。
3. 改造账号详情页，完成详情展示、资料编辑、删除/恢复、授权尝试、授权验证和可用性检查。
4. 为前端开发环境补充 `/api` 代理，使账号模块可与本地后端直接联调。
5. 通过页面联调和前端测试确认账号模块可在“其他模块仍为 mock”的条件下独立运行。

## Open Questions

- 详情页中“授权尝试 + 验证授权”是否需要在 UI 上拆成两个显式步骤，还是以一个向导式表单承载更合适。
- 对于二维码授权区，本轮是保留静态占位卡片，还是直接隐藏整个区域，哪种更符合当前产品预期。
- 账号列表页的平台统计卡，是否需要先按“当前列表结果聚合”呈现，还是在 UI 上明确标注为“当前筛选结果”。
