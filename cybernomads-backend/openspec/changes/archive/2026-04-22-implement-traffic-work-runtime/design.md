## Context

当前后端骨架中 `src/modules/traffic-works/` 仍为空，而引流工作域在产品层面已经是工作区的核心对象。结合领域文档与架构文档，可以确定这部分运行时必须同时承接三类职责：
- 管理工作绑定关系与生命周期状态
- 对外提供列表/详情读模型
- 在创建和更新时触发一次“工作级上下文准备”编排

同时，当前仓库里的策略域、对象绑定域、任务域都还没有完整运行时实现，所以引流工作运行时必须刻意保持边界，不把这些未落地的域强行揉进来。

## Goals / Non-Goals

**Goals:**
- 实现引流工作后端模块的最小完整闭环。
- 支撑当前 MVP 所需的工作创建、查看、状态流转和上下文准备状态回写。
- 保持与 Agent 服务域、产品域、未来策略/对象域的解耦。

**Non-Goals:**
- 不实现任务执行器、任务重试、任务失败恢复和日志明细系统。
- 不在本 change 中展开策略域或对象域的复杂校验实现。
- 不实现前端页面。

## Decisions

### 1. 引流工作运行时采用“元数据 + 工作上下文引用”双语义实现

运行时聚合同时保存：
- 元数据语义：`productId`、`strategyId`、对象绑定、工作名称、生命周期状态
- 上下文语义：当前上下文准备状态、当前准备结果引用

这样能严格对应领域设计文档中的“引用关系”和“运行快照”双语义，而不会退化成单纯 CRUD。

### 2. 创建和更新都内建一次上下文准备编排

创建引流工作与更新引流工作都不是纯数据库写入，而是要在应用服务层编排一次：
- 汇集产品 / 策略 / 对象绑定输入
- 调用 Agent 服务统一入口发起任务规划或等价准备动作
- 写回 `prepared` 或 `failed`

这部分仍然属于引流工作域，因为它表达的是“工作是否具备可运行上下文”，不是任务域所有权。

### 3. 暂时把对象绑定作为引流工作输入，而不是资源域强校验责任

在对象域、策略域运行时尚未完整前，本 change 假定引流工作接收到的对象绑定集合已经满足上层输入要求。运行时会保存并返回它，但不会在这里承担复杂对象解析或跨域一致性判定。

这样做的代价是第一版校验不够深，但好处是不会让引流工作模块依赖尚未落地的其他模块。

### 4. 启动/暂停/更新/终止都围绕明确状态机实现

运行时要固定这些状态语义：
- `ready + prepared` 才能启动为 `running`
- 暂停从 `running` 回到 `ready`
- 更新只允许在非 `running` 状态下发生，并触发重新准备
- `ended` / `archived` / `deleted` 是三个不同的终止语义

这样后续无论前端还是调度器接入，都不会重新解释状态机。

## Risks / Trade-offs

- [风险] 在策略域与对象域未完成前，引流工作运行时的输入校验深度有限。
  → 在本 change 中把这些输入视作上层已解析结果，先把工作域闭环建起来。
- [风险] 创建/更新时引入 Agent 准备编排，会让模块实现比普通 CRUD 更复杂。
  → 这是领域本质要求，不能为简单而去掉。
- [风险] 如果未来工作上下文目录结构调整，当前准备结果引用模型需要扩展。
  → 运行时只保存“上下文引用”语义，不把具体文件路径写进聚合模型。

## Migration Plan

该 change 依赖引流工作契约先存在。运行时落地时应先按 `docs/api/traffic-works.yaml` 与 `docs/sql/traffic-works.sql` 建立适配，再接入 HTTP server。若实现中发现契约缺口，应先回补契约 change，而不是直接偏离实现。

## Open Questions

- 工作上下文准备时，第一版是直接同步等待 Agent 返回，还是先写 `pending` 后异步回写；在当前 MVP 下可先采用更简单的同步编排语义。
