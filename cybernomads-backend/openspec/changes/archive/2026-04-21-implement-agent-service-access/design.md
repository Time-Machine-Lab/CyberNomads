## Context

Cybernomads 当前采用“前端 + 后端 + 外部 Agent 服务”的整体结构。系统在初始化后需要选择并连接一个外部 Agent 服务，当前 MVP 实际只接入一个服务入口，但这个入口需要保持 provider 可替换。领域设计已经明确: Agent 服务接入域负责当前服务配置、连接校验、能力准备以及统一对外访问抽象，而任务规划和任务执行编排仍属于上层应用行为。

当前代码中 `src/modules/agent-access/` 还没有真正实现，因此后端还不具备下列能力:
- 管理当前 Agent 服务连接配置
- 校验外部 Agent 服务是否可达
- 记录连接状态与能力准备状态
- 向上层暴露统一的 Agent 服务访问抽象

本提案聚焦把这些运行时能力落成独立模块，并保持对 OpenClaw 等 provider 的解耦。

## Goals / Non-Goals

**Goals:**
- 实现 Agent 服务接入模块的核心运行时能力。
- 按单激活服务模型持久化当前 Agent 服务配置与状态。
- 通过 provider 抽象向上层暴露统一接入口，供上层发起任务规划请求和任务执行请求。
- 把 HTTP controller、应用服务、状态存储与 provider 端口串起来。
- 为后续 OpenClaw 适配器预留清晰的实现插槽。

**Non-Goals:**
- 不实现 OpenClaw 网关的具体 HTTP 调用细节。
- 不负责任务拆分算法、调度线程和 subagent 内部生命周期。
- 不实现自动重试、断线恢复、容灾切换和多服务管理。
- 不扩展到平台执行脚本和日志观测细节。

## Decisions

### 1. 采用“模块运行时 + provider 端口”分层

**Decision**
- `src/modules/agent-access/` 负责领域对象、应用服务与 controller 协调，provider 相关调用通过端口抽象下沉到适配层。

**Rationale**
- 这样可以确保上层模块只依赖统一抽象，而不依赖 OpenClaw 私有行为。
- 后续切换到远端 SaaS Agent 或自研 Agent 时，主要变更集中在实现层。

**Alternatives Considered**
- 在模块内部直接引用 OpenClaw client。
  - 放弃原因: 会破坏领域边界并提高替换成本。

### 2. 把“任务规划请求 / 任务执行请求”建模为上层发起的统一服务调用

**Decision**
- Agent 服务接入模块不拥有任务规划逻辑本身，但会向上层提供统一的服务调用入口，以便上层把规划请求和执行请求交给当前激活 provider。

**Rationale**
- 这与领域设计保持一致: 规划与执行是上层编排，Agent 接入模块负责统一转发和屏蔽 provider 差异。

**Alternatives Considered**
- 完全不在接入模块中暴露规划/执行入口。
  - 放弃原因: 上层会被迫直接依赖 provider 适配器，失去接入域的价值。

### 3. 状态采用“连接状态 + 能力准备状态”双状态模型

**Decision**
- 运行时仅维护两个核心业务状态: 当前连接状态与当前能力准备状态。

**Rationale**
- 这已经覆盖 MVP 必要信息，并与领域文档和用户确认内容保持一致。
- 不额外引入复杂状态机，降低实现成本。

**Alternatives Considered**
- 引入更细粒度的可执行状态、降级状态和修复中状态。
  - 放弃原因: 超出 MVP 范围。

### 4. 配置变更直接影响现有工作, 但系统只负责暴露事实

**Decision**
- 当用户修改 Agent 服务连接信息或连接中断时，运行时模块只更新状态并向上层暴露影响，不尝试自动迁移或恢复已有工作。

**Rationale**
- 这是当前用户明确接受的 MVP 边界，也能显著降低状态同步复杂度。

**Alternatives Considered**
- 在模块内引入自动暂停、自动恢复或回滚逻辑。
  - 放弃原因: 与当前实现难度和范围不匹配。

## Risks / Trade-offs

- [统一接入口会让模块承担部分转发职责] -> 保持只做 provider 解耦和状态管理，不吸收上层编排语义。
- [单服务模型限制首期灵活性] -> 这是有意收敛的 MVP 范围，后续若扩展多服务可在端口和聚合模型上演进。
- [配置变更会影响已有工作但无自动修复] -> 通过明确状态与错误回传提醒用户处理，避免伪装成系统已恢复。
- [状态与 provider 实际状态可能短暂不一致] -> 当前阶段接受以显式校验和显式能力准备为准，不引入后台保活任务。

## Migration Plan

1. 先确认 Agent 服务 API/SQL 顶层契约已经落定。
2. 在 `src/modules/agent-access/` 中实现领域对象、应用服务、errors 和 controller。
3. 在 `src/ports/` 中定义统一的 provider 访问端口与状态持久化端口。
4. 接入当前的存储适配层并暴露 HTTP 入口。
5. 为 OpenClaw 适配器提供对接点，再由后续 provider 提案补齐具体实现。

## Open Questions

- 凭证在运行时模块中以何种方式保存和返回给前端，需要与实际安全策略进一步对齐。
- 当前前端是否需要立即展示“能力准备进行中”与“能力准备失败”的细分态，还是只先展示结果态，需要在实现前确认展示策略。
