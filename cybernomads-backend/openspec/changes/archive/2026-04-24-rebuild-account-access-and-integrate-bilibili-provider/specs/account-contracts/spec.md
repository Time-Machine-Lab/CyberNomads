## MODIFIED Requirements

### Requirement: Account contracts SHALL define wrapper-account identity separately from resolved platform identity
The system SHALL define top-level account contracts around a stable wrapper account identified by `accountId`, while treating resolved platform identity as optional profile data observed after verification rather than as the account's business key.

#### Scenario: Wrapper account identity does not require resolved platform UID at creation time
- **WHEN** account creation contracts are defined for the rebuilt model
- **THEN** the contracts SHALL require a stable `accountId` generated by the system
- **AND** the contracts SHALL NOT require `resolvedPlatformAccountUid` during creation

#### Scenario: Resolved platform identity may repeat across wrapper accounts
- **WHEN** top-level contracts define resolved platform profile fields
- **THEN** the contracts SHALL allow multiple wrapper accounts on the same platform to carry the same `resolvedPlatformAccountUid`
- **AND** they SHALL treat that field as observed platform data instead of as a uniqueness key

### Requirement: Account contracts SHALL define connection-centric account state and diagnostic availability separately
The system SHALL define wrapper-account state primarily from lifecycle status and connection status, while keeping availability as a secondary diagnostic observation rather than as the main access-workflow gate.

#### Scenario: New account starts as not logged in but remains fully manageable
- **WHEN** a wrapper account is created without a verified credential
- **THEN** the contracts SHALL represent it with a first-class `not_logged_in` connection status
- **AND** the account SHALL remain readable and editable

#### Scenario: Availability remains diagnostic
- **WHEN** top-level contracts expose availability information
- **THEN** the contracts SHALL describe it as diagnostic runtime context
- **AND** they SHALL NOT require a separate availability-check action in order to complete the basic access-session flow

### Requirement: Account contracts SHALL publish account-bound access-session contracts
The system SHALL publish dedicated API and SQL contracts for account-bound access sessions, covering manual token submission, QR session start, QR polling, verification, and log retrieval.

#### Scenario: Access-session API is published separately from account CRUD
- **WHEN** top-level API contracts define account access workflows
- **THEN** the system SHALL publish a dedicated access-session controller under the account resource
- **AND** account CRUD routes SHALL remain focused on the stable wrapper account object

#### Scenario: Account detail keeps session payload light
- **WHEN** account detail contracts include information about ongoing access work
- **THEN** the contracts SHALL expose at most a summary of the current or most recent session
- **AND** full session detail and logs SHALL be readable through session-scoped contracts
