## Context

当前账号模块已经具备稳定账号对象、当前生效凭证、待验证授权尝试、可用性检查等一期运行时能力，但新增账号仍主要依赖 `POST /api/accounts` 直接提交 `platformAccountUid` 来创建对象。这个入口与当前产品目标不一致，因为真实平台接入更接近“先通过脚本拿到令牌，再解析平台身份，再决定是否新建或恢复账号”的流程。

前端现状也放大了这个错位：

- 列表页没有真正的新增入口，只能查看已有账号。
- 详情页把“直接凭证接入”和“扫码授权”拆成两块，并继续把 `token_input`、`cookie_input`、`qr_authorization` 直接暴露给用户。
- 二维码区域目前仍是占位，而后端真实 challenge 尚未通过账号模块 HTTP 契约发布。

后端现状同样存在两个关键约束：

- `src/modules/accounts/service.ts` 当前创建路径要求先给出 `platformAccountUid`，不适合“令牌优先”的新增流程。
- `src/ports/account-platform-port.ts` 当前接口主要围绕“已有账号 + 授权尝试 + 验证”建模，尚未覆盖“无稳定账号对象时的令牌解析与身份识别”。

同时，领域设计文档已经明确了几个不能退让的边界：

- 稳定账号对象、当前生效凭证、待验证授权尝试必须隔离。
- 普通详情只返回脱敏管理视图，不能直接返回原始凭证。
- 同一 `platform + platformAccountUid` 只能对应一个稳定账号对象，被逻辑删除后应恢复原对象。
- 平台差异必须收敛在平台能力抽象后，由脚本适配层实现，而不是扩散到账号模块和前端页面。

因此，这次设计需要在不推翻现有账号聚合语义的前提下，引入一个独立的“账号接入会话”资源，专门承载新增账号前的令牌接入和身份解析流程。

## Goals / Non-Goals

**Goals:**

- 为新增账号提供“令牌优先”的接入流程，并把扫码和手工录入统一到同一套会话语义下。
- 保持 `PlatformAccount` 聚合语义稳定，不为了新增流程引入“半成品账号对象”。
- 允许已有账号在详情页通过同一块“令牌接入”区域发起换令牌流程，同时继续遵守“验证成功后才切换当前生效凭证”的约束。
- 扩展平台能力抽象，让平台脚本既能在无既有账号对象时解析身份，也能在已有账号上执行换令牌与可用性检查。
- 明确顶层 API/SQL 文档、后端模块边界和前端页面改造落点，形成可直接进入实现的需求文档。

**Non-Goals:**

- 不在本轮实现真实平台脚本，只定义平台脚本需要遵守的统一接口语义。
- 不在本轮开放原始令牌回显、令牌历史、终端日志等敏感或未成熟能力。
- 不在本轮推翻现有账号详情的脱敏边界，也不把普通详情改造成原始凭证读取接口。
- 不要求删除现有直接创建账号能力；它可以保留为补充路径，但新增账号主流程转向接入会话。
- 不在本轮追求最终 UI 视觉稿，只定义前后端对接所需的页面能力。

## Decisions

### 1. 引入独立的 `AccountOnboardingSession` 资源，而不是先创建“未识别账号”

**Decision**

- 新增一个独立于 `PlatformAccount` 聚合的账号接入会话资源，用于承载：
  - 平台选择
  - 接入方式选择（扫码或手工令牌录入）
  - challenge 摘要
  - 待解析输入引用
  - 解析出的平台身份与资料摘要
  - 候选令牌引用
  - 最终新建/恢复结果
- 接入会话在稳定账号对象创建前存在，成功完成后被标记为已消费或已关闭。

**Rationale**

- 领域设计已经说明稳定账号对象的业务身份是 `platform + platformAccountUid`。如果先创建“没有 UID 的账号”，会让聚合根在最关键的身份语义上失真。
- 接入会话天然属于短生命周期过程状态，与长期稳定账号对象的职责不同，单独建模更清晰。

**Alternatives Considered**

- 先创建一个 `platformAccountUid` 为空的账号对象，再等待后续验证补齐身份
  - 放弃原因：会破坏稳定身份约束，也会让“恢复原对象”语义变得模糊。
- 直接把新增账号的暂存状态塞进 `platform_accounts.authorization_attempt_*`
  - 放弃原因：该槽位属于既有账号聚合内的授权尝试，不适合承载“尚未存在稳定账号对象”的状态。

### 2. 用户侧统一叫“令牌”，内部仍保留凭证抽象

**Decision**

- 对前端页面、HTTP 文案和新增流程，用统一业务术语“令牌接入 / 更换令牌”。
- 对运行时和平台适配层，仍保留 `credential` 这一内部抽象，允许平台脚本返回结构化 JSON payload，而不是强制收窄成单一字符串。

**Rationale**

- 用户关心的是“拿到平台可用的令牌并能被系统消费”，而不是 token、cookie、session 的细分名词。
- 平台脚本底层很可能返回多字段材料；如果契约层过早强制所有平台都只返回单字符串 token，后续会反过来污染适配层设计。

**Alternatives Considered**

- 前后端所有层都统一改成单字符串 token
  - 放弃原因：实现简单，但会让平台差异重新回流到账户模块与脚本适配层。
- 继续把 token / cookie / session 全部显式暴露给页面
  - 放弃原因：不符合本轮产品简化目标，也让页面承担了不必要的实现细节。

### 3. 新增账号与已有账号换令牌走两条运行时路径，但共用同一套平台能力语义

**Decision**

- 新增账号走“账号接入会话”路径：
  - 发起接入会话
  - 获取 challenge 或提交令牌
  - 解析平台身份与候选令牌
  - 基于已验证结果完成新建/恢复
- 已有账号换令牌继续走账号聚合内的授权尝试路径：
  - 发起授权尝试
  - 返回待验证 attempt 摘要与可选 challenge
  - 解析成功后再切换当前生效凭证
- 两条路径共用同一个平台能力抽象和统一脚本输入输出语义。

**Rationale**

- 新增账号和已有账号换令牌的领域语义不同：前者还没有稳定账号对象，后者必须保护现有当前生效凭证。
- 但两者底层都依赖平台脚本获取 challenge、解析身份、产出候选令牌，复用统一平台能力最稳妥。

**Alternatives Considered**

- 用一个巨大的通用接口同时处理新增和换令牌
  - 放弃原因：会把“有账号”和“无账号”的状态机混在一起，契约和实现都更难维护。

### 4. 顶层契约拆成“已有账号 Controller + 接入会话 Controller”，并新增独立会话表

**Decision**

- 保持已有账号 Controller 继续承载：
  - 列表、详情、资料更新、删除、恢复
  - 已有账号的授权尝试、验证、可用性检查
- 新增账号接入会话 Controller，单独发布接入会话 API。
- SQL 层新增 `account_onboarding_sessions` 表，并用独立 `docs/sql/account_onboarding_sessions.sql` 描述。
- `platform_accounts` 表仍然承载稳定账号对象与聚合内授权尝试，不把新增账号前的临时状态塞回原表。

**Rationale**

- 这符合“一份 `.yaml` 对应一个 Controller，一张表对应一个 `.sql` 文件”的项目约束。
- 会话表持久化后，二维码 challenge、待解析状态、验证失败原因等都能在服务重启后保留，稳定性更好。

**Alternatives Considered**

- 把接入会话也塞进 `docs/api/accounts.yaml` 对应的同一个 Controller
  - 放弃原因：资源语义不同，路由和 DTO 也会明显膨胀。
- 只把会话放内存，不落库
  - 放弃原因：会让扫码和轮询类流程在进程重启后全部丢失，不符合高可用和稳定性目标。

### 5. 平台能力扩展为“开始接入 / 解析令牌 / 验证替换 / 可用性检查”四类统一动作

**Decision**

- 平台适配层在 provider-neutral 接口下新增或拆分以下能力语义：
  - 开始令牌接入并返回 challenge 摘要
  - 在无既有账号对象时解析令牌并产出身份、资料、候选令牌
  - 在已有账号上验证待替换令牌并产出规范化结果
  - 基于当前生效令牌执行可用性检查
- 账号模块只消费规范化结果，不感知脚本函数名、脚本参数差异或底层实现细节。

**Rationale**

- 当前 `AccountPlatformPort` 只适合已有账号场景，不能优雅承载“无 stable account snapshot”的令牌解析。
- 把平台差异继续压在 port/adapters 内部，才能避免新增流程再次污染 controller 和 page 层。

**Alternatives Considered**

- 让账号模块直接调用不同平台脚本名，例如 `getBilibiliQrCode()` / `resolveBilibiliToken()`
  - 放弃原因：这会直接破坏平台抽象边界。

### 6. 前端采用“列表页新增入口 + 独立接入页 + 详情页统一令牌区”的页面结构

**Decision**

- 列表页新增真实“新增账号”入口，跳转到接入页。
- 新增账号接入页负责：
  - 平台选择
  - 接入方式选择
  - challenge 展示
  - 令牌解析结果确认
  - 最终新建/恢复
- 详情页把“直接凭证接入”和“扫码授权”合并为同一块“令牌接入”区域，用于已有账号更换当前令牌。

**Rationale**

- 新增账号和已有账号详情不适合再共用同一个页面入口。
- 详情页应该围绕“已存在稳定账号对象”的管理语义；新增账号页则是接入流程。

**Alternatives Considered**

- 仍然只保留详情页，在空白账号详情中承载新增流程
  - 放弃原因：会导致路由语义模糊，也不利于处理“尚无 accountId”的状态。

## Risks / Trade-offs

- [新增接入会话表和 Controller，会让账号模块边界比当前更宽] → 通过独立资源命名、独立顶层文档和独立 service 边界控制复杂度，不把会话逻辑反向塞回 `PlatformAccount` 聚合。
- [接入会话与已有账号授权尝试并存，状态机会出现两条路径] → 明确一条用于“无账号对象”的接入，一条用于“已有账号”的替换；两者只共享平台能力，不共享存储槽位。
- [对外统一叫令牌，但内部仍保留凭证抽象，可能让实现层出现两套命名] → 规定“页面/HTTP 文案用令牌，内部 service/port 用 credential”，并在 DTO mapper 层做统一映射。
- [接入会话持久化后，候选令牌引用需要妥善清理] → 在 finalize、expire、cancel 路径上明确关闭/消费语义，并限制普通查询读取原始引用目标。
- [若解析出的身份已对应一个现存账号，新增流程会出现‘不是新建而是跳转到已有账号’的结果] → 在会话 finalize 结果中明确返回 `existingAccountId` 与建议动作，让前端可以稳定跳转。

## Migration Plan

1. 先在 OpenSpec 中补齐接入会话 capability、账号 contracts/runtime/platform/frontend 的 delta specs。
2. 更新顶层 API/SQL 文档：
   - 修改 `docs/api/accounts.yaml`
   - 新增 `docs/api/account-onboarding-sessions.yaml`
   - 新增 `docs/sql/account_onboarding_sessions.sql`
3. 后端新增接入会话模块及持久化实现，再调整账号模块既有授权尝试返回 challenge 的行为。
4. 扩展平台能力 port 和 stub adapter，使其同时覆盖新增账号接入与已有账号换令牌。
5. 最后改造前端列表页、接入页和详情页，完成统一令牌接入的联调。

## Open Questions

- 接入会话在前端是否采用单独页面 `/accounts/new`，还是后续会演进为弹层/抽屉；本次只约束需要独立入口，不强制最终展示形式。
- QR challenge 对前端的最小契约是否只需要图片地址/文本摘要，还是需要同时支持轮询 token 状态；本次 spec 可以先要求 provider-neutral challenge，具体字段在 API 文档阶段细化。
- 直接 `POST /api/accounts` 的传统人工创建能力在最终产品里是否继续对用户可见；本次设计保留为后备能力，但新增主流程不再依赖它。
