## Context

策略领域设计已经明确了策略模块的核心边界：策略是独立的 Markdown 提示词资产，结构化元数据与正文内容分离，正文支持通过内嵌注释标记表达“整篇快照插入”的来源信息，并通过统一占位符语法声明后续对象绑定需求。

当前缺失的不是更多领域讨论，而是顶层契约。后端还没有 `docs/api/strategies.yaml` 与 `docs/sql/strategies.sql`，这意味着后续实现很容易重新被前端草案牵着走，或把编辑器辅助能力直接做成不稳定的 REST 接口。

本次设计需要回答三个实际问题：
- 策略模块的公开 API 最小应暴露什么能力。
- 哪些内容应进入结构化 SQL 契约，哪些应继续留在 Markdown 正文或模块内部能力中。
- 导入块标记、占位符解析和摘要生成这些“派生语义”应如何优雅落在契约层，而不把 MVP 做重。

## Goals / Non-Goals

**Goals:**
- 为策略模块建立与领域设计一致的顶层 API 与 SQL 契约。
- 明确策略模块 MVP 只提供创建、更新、列表、详情四类公开 API。
- 明确策略最小元数据集合，包括 `strategyId`、`name`、`summary`、`tags`、`contentRef`、`createdAt`、`updatedAt`。
- 明确 Markdown 正文继续通过 `content_ref` 与文件系统原文关联，不把完整正文或导入块拆成额外关系表。
- 明确对象占位符与导入块标记在契约中的语义边界，为后续实现留出稳定扩展点。

**Non-Goals:**
- 不定义策略模块运行时实现细节。
- 不新增独立的“整篇插入来源策略” HTTP 接口。
- 不定义公开的“编译策略输入”或“提取占位符” REST 接口。
- 不引入删除、草稿、发布、归档、版本链、实验统计或运行效果字段。
- 不定义对象绑定路由、任务规划或引流工作联动契约。

## Decisions

### 1. 公开 API 维持和产品模块同构的极简骨架

**Decision**
- `docs/api/strategies.yaml` 只定义创建、更新、列表、详情四类公开接口。

**Rationale**
- 这与产品模块已经建立的契约模式一致，便于后续实现按相同骨架推进。
- 策略的“整篇插入”本质上是编辑器对 Markdown 正文的加工，而不是独立的后端业务事务。
- 先稳定最小 API 面，再把编译和解析等能力留给模块内部或后续上层应用服务，会让契约更耐用。

**Alternatives Considered**
- 为“整篇插入”新增独立 REST 接口
  - 放弃原因：会把编辑器行为误建模为服务边界，并让后端提前承担不必要的正文拼装职责。
- 直接把“编译策略输入”做成公开 REST 接口
  - 放弃原因：当前主要是模块内部消费能力，还不适合过早暴露。

### 2. SQL 继续采用“元数据在 SQLite，正文在文件系统”的薄领域模式

**Decision**
- `docs/sql/strategies.sql` 只定义结构化元数据与 `content_ref`，完整 Markdown 正文继续通过文件系统原文承载。

**Rationale**
- 这与架构文档、后端规范和产品模块现有做法一致，避免策略模块单独走出一条不同存储路线。
- 导入块标记本身就是 Markdown 注释，天然适合和正文一起存储，不需要再拆额外表。

**Alternatives Considered**
- 将完整 Markdown 正文直接内嵌到 SQLite 表
  - 放弃原因：会打破当前“元数据 / 原文分离”的既有模式。
- 为导入块单独建立关系表
  - 放弃原因：导入块只承担编辑辅助语义，不值得在 MVP 引入额外结构化建模成本。

### 3. 策略详情优雅暴露元数据与高价值派生信息，但不暴露导入来源列表

**Decision**
- 详情模型返回 `tags` 以及从正文解析得到的 `placeholders`。
- 不新增独立的 `importedSources` 字段，导入来源继续由正文中的注释标记承载。

**Rationale**
- `tags` 本身就是结构化元数据，应直接对外暴露。
- `placeholders` 对编辑器、校验和后续上层消费都具有直接价值，作为详情派生字段很自然。
- `importedSources` 的唯一价值是帮助编辑器识别高亮区块，而这已经能通过 Markdown 注释本身完成；重复暴露一个单独字段只会制造双重真相源。

**Alternatives Considered**
- 详情只返回纯正文，不返回任何派生字段
  - 放弃原因：会让前端和后续上层重复做相同解析。
- 详情同时返回 `importedSources`
  - 放弃原因：与正文标记重复，且后续容易与正文真实内容不一致。

### 4. `summary` 采用“显式优先，缺省派生”的优雅降级策略

**Decision**
- 创建和更新请求允许显式传入 `summary`。
- 若调用方未传 `summary`，后续运行时可以从正文自动派生策略摘要。

**Rationale**
- 这兼顾了人工可控摘要与低负担录入体验。
- 列表查询需要稳定摘要，但不应强迫每个用户都手工维护。

**Alternatives Considered**
- `summary` 必填
  - 放弃原因：增加录入负担，和提示词资产的实际使用习惯不匹配。
- `summary` 永远自动派生，不允许显式传入
  - 放弃原因：会限制策略作者对摘要展示的控制力。

### 5. 对象占位符语法在契约层固定为 `{{objectType:objectKey}}`

**Decision**
- API 和规格文档统一采用 `{{objectType:objectKey}}` 作为对象引用的规范语法。

**Rationale**
- 这种写法清晰、稳定、易解析，也避免把展示名误当作绑定主键。
- 它与模板系统常见写法相近，后续无论编辑器高亮还是编译阶段净化都更可控。

**Alternatives Considered**
- 继续使用 `@[账号A]` 一类展示名写法
  - 放弃原因：重命名不稳定，也不利于机器解析。
- 使用无类型占位符 `{{ref:key}}`
  - 放弃原因：会推迟类型语义，后续绑定校验的表达力不足。

## Risks / Trade-offs

- [详情返回 `placeholders` 会引入派生字段] → 通过明确其来源于正文解析并避免把它反向作为持久化真相源来降低一致性风险。
- [导入块只存于 Markdown 注释，后续正文可能被手工破坏] → 接受这是正文级编辑自由的一部分，后续实现只把它当“尽力识别”的辅助标记，不当作强事务结构。
- [`summary` 既可显式传入又可自动派生，行为存在双路径] → 通过在 API 契约中明确“显式优先、缺省派生”的规则降低歧义。
- [不公开编译接口可能让部分前端能力暂时拿不到后端支持] → 当前优先稳定领域与契约边界，后续如确有需要再以新变更公开对应接口。

## Migration Plan

1. 基于当前策略领域设计，新增 `openspec/specs/strategy-contracts/spec.md`。
2. 根据规格补齐 `docs/api/strategies.yaml`。
3. 根据规格补齐 `docs/sql/strategies.sql`。
4. 在后续独立变更中，再基于这些顶层契约推进 `strategy-module-runtime` 与 `src/modules/strategies/` 实现。

## Reusable Contract Conclusions

- 最小结构化元数据集合固定为 `strategyId`、`name`、`summary`、`tags`、`contentRef`、`createdAt`、`updatedAt`。
- `content_ref` 的职责固定为“从结构化元数据指向完整 Markdown 原文”，不承载导入块或占位符的拆表语义。
- 公开 REST API 固定为创建、更新、列表、详情四类能力；整篇快照插入不单独建接口。
- 详情模型返回 `tags` 和 `placeholders`，但不返回 `importedSources`；导入来源继续以内嵌 Markdown 注释为真相源。
- 对象占位符规范固定为 `{{objectType:objectKey}}`，后续运行时和对象绑定实现应直接复用这一正文语法。

## Open Questions

- 详情接口中的 `placeholders` 最终是否只返回去重后的稳定键集合，还是返回包含原始片段位置的更丰富结构；当前建议先以最小结构为主。
- 后续若引流工作域需要“可编译策略输入”的公开接口，是否继续保持为内部模块能力，还是升级为单独契约；当前阶段先不展开。
