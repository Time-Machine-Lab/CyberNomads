## Context

账号池领域设计文档已经明确了账号模块的核心语义：`PlatformAccount` 是稳定聚合根，账号模块同时管理当前生效凭证、待验证授权尝试、生命周期状态、授权状态、可用状态，以及逻辑删除与恢复语义。与此同时，当前后端仓库中仍然没有账号模块的任何顶层契约文档，也没有与之对应的运行时规格说明。

现有后端已经形成了两种可复用模式：
- 产品模块提供了“薄领域 + 顶层 API / SQL 契约先行”的模式。
- Agent Access 模块提供了“状态驱动 + 敏感凭证不直接暴露 + 文件系统承载 secret payload”的模式。

账号模块的后端设计不能简单复制其中任何一个，而应结合两者：
- 在契约层保持像产品模块那样清晰的 API / SQL 单一真理。
- 在运行时语义上保持像 Agent Access 那样的验证、切换、受控解析和敏感数据隔离。

当前还存在三个明确约束：
- 账号详情必须是脱敏管理视图，不能直接暴露可消费的原始凭证。
- 当前生效凭证与待验证授权尝试必须严格分离。
- 三维状态不能再被收敛回单一 `status` 字段，且必须补上最低消费条件。

## Goals / Non-Goals

**Goals:**
- 定义账号模块的顶层 API 契约与 SQL 契约。
- 定义账号模块运行时必须满足的后端行为要求。
- 将账号领域设计中的关键决策映射为可执行的后端设计约束。
- 明确账号详情、受控凭证解析、授权尝试、当前凭证切换和状态消费规则的边界。
- 为后续账号模块实现提供稳定输入，避免实现阶段反向定义接口和数据结构。

**Non-Goals:**
- 不实现账号模块代码。
- 不实现平台脚本、扫码协议、凭证加密和 secret 存储细节。
- 不定义观测终端日志、验证日志明细和调试控制台能力。
- 不设计工作区绑定、策略角色映射或下游任务执行语义。
- 不以当前前端 mock 结构作为后端契约来源。

## Decisions

### 1. 先定义顶层契约与运行时规格，再进入模块实现

**Decision**
- 本次变更先产出 `docs/api/accounts.yaml`、`docs/sql/accounts.sql` 以及账号模块运行时规格文档，不进入代码实现。

**Rationale**
- 这符合项目中 `docs/` 是唯一真理的治理约束。
- 账号模块的状态和凭证语义明显比产品模块复杂，若先写代码，极易把前端 mock 或页面表达误当后端边界。

**Alternatives Considered**
- 先实现账号模块代码，再补顶层文档
  - 放弃原因：会违反项目规则，也会放大契约漂移风险。

### 2. 账号模块按“稳定账号对象 + 当前凭证 + 待验证授权尝试”建模

**Decision**
- 后端设计将 `PlatformAccount` 视为稳定账号对象，并明确区分：
  - 当前生效凭证
  - 待验证授权尝试

**Rationale**
- 这直接承接了账号领域设计文档中修正后的核心约束。
- 能避免 `qr_authorization` 这类待验证流程产物提前污染当前生效凭证。

**Alternatives Considered**
- 将扫码授权、Token 提交、Cookie 提交全部并入当前凭证字段
  - 放弃原因：与“单当前凭证 + 验证后切换”冲突，且无法表达授权尝试中间态。

### 3. 账号详情与凭证消费能力分层

**Decision**
- 顶层 API 契约中的账号详情接口只返回脱敏管理视图。
- 当前生效凭证解析能力属于运行时受控能力，不作为普通前端接口暴露。

**Rationale**
- 这符合账号领域文档中“详情脱敏 + 受控解析”的边界。
- 能防止普通详情接口、日志体系和页面状态查询意外拿到凭证 payload。

**Alternatives Considered**
- 让账号详情接口直接承载当前凭证内容
  - 放弃原因：会把管理视图与敏感能力混层，后续很难收敛安全边界。

### 4. 结构化状态存 SQLite，敏感 payload 用引用方式隔离

**Decision**
- 顶层 SQL 文档只定义账号元数据、状态、引用关系和查询字段。
- 当前生效凭证 payload 与授权尝试 payload 以引用语义存在，不作为 SQL 正文存储内容暴露。

**Rationale**
- 这与产品模块“结构化状态 + 内容引用”的做法一致，也与 Agent Access 模块“state in SQLite, secret in file store”的模式一致。
- 能让 SQL 契约保持最小结构化表达，不被敏感字段和协议细节污染。

**Alternatives Considered**
- 将 token / cookie 等原始 payload 直接写入 SQLite 表
  - 放弃原因：会提高泄露风险，也会让顶层 SQL 契约和运行时 secret 管理混层。

### 5. 三维状态保持独立，但必须定义最低消费条件

**Decision**
- 生命周期状态、授权状态、可用状态继续作为独立维度存在。
- 同时在运行时规格中明确账号进入可消费集合的最低条件。

**Rationale**
- 仅强调状态独立不够，后续实现仍然无法判断账号是否能被下游使用。
- 明确最低消费条件后，后端服务和未来下游模块才能围绕统一规则实现。

**Alternatives Considered**
- 将三维状态再次压缩为一个 `status`
  - 放弃原因：会丢失领域语义，重新制造账号可读但不可消费这类边界混乱。

### 6. 账号模块当前只支持逻辑删除与恢复，不支持物理删除

**Decision**
- 顶层契约和运行时规格都只定义逻辑删除与恢复能力，不定义物理删除。

**Rationale**
- 账号模块有稳定身份和恢复优先约束，物理删除会立即引入身份复用、引用清理和审计语义问题。
- 当前阶段需要先把对象稳定性做对，而不是追求删除完整性。

**Alternatives Considered**
- 同步定义物理删除
  - 放弃原因：复杂度过高，且与“恢复优先”设计冲突。

## Risks / Trade-offs

- [账号模块同时涉及对象状态、凭证、授权尝试，文档范围天然比产品模块更重] → 通过拆分 `account-contracts` 与 `account-module-runtime` 两个 capability，分别承接契约与行为约束。
- [不把凭证解析做成 HTTP 接口，前端会少一个“直接拿凭证”的捷径] → 接受这个约束，保持管理视图与系统能力分层。
- [SQL 契约采用引用而非直接落敏感 payload，会让实现多一层存储适配] → 接受该复杂度，以换取清晰的安全边界和后续 secret 管理弹性。
- [逻辑删除而非物理删除会让数据长期保留] → 这是当前领域稳定性优先的有意选择，后续如确实需要物理删除，再单独设计变更。

## Migration Plan

1. 基于账号领域设计文档定义账号模块 proposal / design / specs / tasks。
2. 在 `docs/sql/` 中新增账号模块首版 SQL 契约文档。
3. 在 `docs/api/` 中新增账号模块首版 API 契约文档。
4. 在 `openspec/specs/` 中新增账号模块契约与运行时能力规格。
5. 后续若进入实现阶段，再以本次文档为输入补充代码设计和实现任务。

## Open Questions

- 当前阶段是否允许一个账号同时存在多条历史授权尝试记录，还是只保留一条最近尝试。
- 顶层 API 是否需要显式暴露“可用性检查原因”字段，还是仅返回状态与摘要原因。
- `platform_metadata` 在首版契约中是否需要限制为对象结构，还是保留更宽松的 JSON 语义。
