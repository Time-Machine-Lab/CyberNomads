## Context

策略模块当前已经有领域设计文档、首版 API/SQL 契约和一份已归档的 `strategy-contracts` OpenSpec 能力，但还没有任何运行时代码。更关键的是，现有文档仍把正文中的特殊标记表述为“对象引用占位符”，而本次需求已经明确：

- 策略的本质是一份 Markdown 模板。
- 策略之间没有活引用关系，左侧插入只是把其他策略正文整篇拷贝到当前正文中。
- 后端不负责高亮、位置、编辑器渲染，也不负责真正的资源绑定。
- 当前版本的占位符只是一种参数声明，字段固定为 `type`、`key`、`defaultValue`。
- 当前只支持 `string` 与 `int` 两种参数类型。

这意味着策略模块的实现重点不再是“编译并绑定对象”，而是“稳定地保存模板、解析参数声明、向其他模块暴露一致契约”。实现会跨越模块代码、HTTP 接口、SQLite 元数据存储、Markdown 正文落盘和测试，因此需要单独设计。

## Goals / Non-Goals

**Goals:**
- 在后端实现策略模块的领域对象、应用服务、HTTP Controller 和持久化适配。
- 支持创建、更新、列表和详情四个核心能力。
- 让策略详情返回完整 Markdown 正文和去重后的参数占位符声明集合。
- 在创建与更新阶段校验占位符语法，保证存量策略正文始终可稳定解析。
- 在 `summary` 缺省时从 Markdown 派生摘要并持久化。
- 保持导入块标记只作为 Markdown 正文的一部分保存，不引入关系表或独立接口。
- 在代码实现前对齐 `docs/design/domain/`、`docs/api/` 和 `docs/sql/` 文档。

**Non-Goals:**
- 不实现策略删除。
- 不实现草稿、发布、归档、版本管理或实验能力。
- 不实现真实资源绑定、对象路由、占位符值收集或模板编译接口。
- 不返回占位符位置信息，也不承担编辑器高亮逻辑。
- 不把导入块标记单独持久化到数据库。

## Decisions

### 1. 采用“SQLite 元数据 + 文件系统 Markdown 正文”的双存储实现

**Decision**
- 策略模块沿用产品模块已经验证过的后端结构：SQLite 负责最小元数据，文件系统负责 Markdown 原文，二者通过 `content_ref` 关联。

**Rationale**
- 这与现有 `docs/sql/strategies.sql` 的薄领域边界一致。
- 元数据更新和正文读写职责分离后，策略编辑和列表查询都更直接。
- 与产品模块保持相似形态，可以降低后续实现和维护成本。

**Alternatives Considered**
- 把 Markdown 正文直接存进 SQLite
  - 放弃原因：会让结构化元数据和大段正文耦合在同一表中，不符合当前项目的存储分层。
- 为导入块、占位符再单独建表
  - 放弃原因：当前语义只要求“正文可解析”，不需要关系化存储。

### 2. 将占位符统一定义为“参数占位符”，并在保存时进行语法校验

**Decision**
- 策略正文中的可填空标记统一使用参数占位符语法：
  - `{{string:title="默认标题"}}`
  - `{{string:cta_text=""}}`
  - `{{int:max_retry=3}}`
- 运行时返回的结构化契约固定为 `type`、`key`、`defaultValue`。
- 同一策略中允许同一个 `key` 在正文中出现多次，但其 `type` 与 `defaultValue` 必须完全一致；一旦冲突则拒绝保存。
- 后端不返回位置信息，只返回去重后的参数声明集合。

**Rationale**
- 这最符合“字段映射”而不是“对象绑定”的真实需求。
- `defaultValue` 必填，可以让前端和后续模块在没有额外配置的情况下也能得到完整模板语义。
- 去重后的声明集合比返回全文位置信息更适合当前版本的消费方式。

**Alternatives Considered**
- 继续沿用 `{{objectType:objectKey}}`
  - 放弃原因：会误导实现继续向资源绑定方向演化，不符合本次最终确认的范围。
- 用数据库单独持久化占位符定义
  - 放弃原因：正文已经是单一事实来源，再额外持久化会引入一致性成本。
- 允许任意文本默认值且不校验
  - 放弃原因：会让详情接口无法稳定返回类型化声明。

### 3. 对 `summary` 采用“写入时回填”而不是“读取时临时生成”

**Decision**
- 如果调用方未显式提供 `summary`，策略服务在创建或更新时从 Markdown 派生摘要，并将结果写回元数据表。
- 摘要派生规则保持简单稳定：跳过空行和导入注释，提取第一段非空文本，做基础 Markdown 符号清理和长度截断。

**Rationale**
- 列表接口需要稳定摘要，写入时回填可以避免每次列表查询都重新扫描全文。
- 显式摘要仍可覆盖自动派生，给调用方保留控制权。

**Alternatives Considered**
- 永远要求前端显式传 `summary`
  - 放弃原因：与现有契约里“可省略”的规则不一致。
- 在列表查询时动态生成摘要
  - 放弃原因：增加读取成本，也会让同一策略在不同时间可能生成不同摘要。

### 4. 导入块标记继续作为原文语义透传，不进入运行时关系模型

**Decision**
- 类似 `<!-- cn-strategy-import:start source-id="..." -->` 的导入块标记跟随 Markdown 原文一起保存和返回。
- 策略模块不新增“快照插入接口”、不建立导入关系表、也不在详情接口中额外返回来源列表。

**Rationale**
- 这与当前“整篇插入只是编辑辅助语义”的约束一致。
- 前端完全可以基于原文标记完成高亮或包装渲染，不需要后端承担更多状态。

**Alternatives Considered**
- 保存独立的 import metadata
  - 放弃原因：会把纯正文语义升级成关系模型，超出 MVP。

### 5. 策略模块只暴露模板与参数声明，不负责运行时填值或编译

**Decision**
- 当前阶段的后端策略模块只暴露：
  - 完整 Markdown 模板
  - 最小元数据
  - 参数占位符声明集合
- 不提供单独的“编译最终字符串”接口，也不接受或保存参数实际值。

**Rationale**
- 你已经明确后端此阶段只需要遵循契约，并把完整字符串模板交给其他模块消费。
- 把“模板管理”与“运行时填值”拆开，边界更清晰，也更容易后续演进。

**Alternatives Considered**
- 在本次实现里直接做模板编译
  - 放弃原因：这会引入值收集、替换、转义和绑定责任，显著扩大范围。

## Risks / Trade-offs

- [旧文档仍使用对象引用语言] → 先更新 `docs/design/domain/`、`docs/api/` 和 `docs/sql/`，再开始代码实现，避免新代码建立在旧契约上。
- [严格的占位符校验会拦住格式不规范的历史正文] → 通过明确语法规则、补充测试和必要的迁移修订降低风险。
- [不返回位置信息会让前端自行做渲染解析] → 当前这正是刻意的边界收敛，前端可直接依据 Markdown 标记自行定位。
- [摘要自动派生规则可能不完全符合所有文案预期] → 保留显式 `summary` 覆盖能力，自动派生仅作为默认回退。
- [不做运行时编译可能让后续消费方多一步处理] → 当前接受这点成本，以换取策略模块职责更清晰、更稳定。

## Migration Plan

1. 先修订 `docs/design/domain/Cybernomads策略领域设计文档.md`，把“对象引用占位符”统一改成“参数占位符”语义。
2. 更新 `docs/api/strategies.yaml`，将占位符结构调整为 `type`、`key`、`defaultValue`，并去掉对象绑定表述。
3. 校验 `docs/sql/strategies.sql`，确认继续保持薄表结构且不新增占位符或导入关系表。
4. 在 `src/modules/strategies/`、`src/adapters/storage/` 和应用启动层实现运行时代码。
5. 增加测试并验证 CRUD、摘要回填、占位符解析/校验与范围排除项。
