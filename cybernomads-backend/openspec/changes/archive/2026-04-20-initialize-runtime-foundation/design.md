## Context

Cybernomads 后端当前仍是启动即空转的骨架，没有运行时基础初始化能力。根据现有架构文档与开发规范，系统启动后应在当前目录下准备 `cybernomads/` 本地运行根目录，目录下包含 `product/`、`strategy/`、`work/` 三个固定子目录；同时还需要创建或打开 SQLite 文件，并通过运行时 SQL 资产完成系统级初始化。

这个变更存在几个明确约束：
- 运行时不能依赖 `docs/sql/`，因为该目录属于开发文档而不是产品发布资产。
- 当前尚未定义正式业务表，因此本次只允许创建系统级初始化表，不创建业务表。
- 如果 `cybernomads/` 已存在，只跳过根目录创建，但 SQLite 与 SQL 脚本执行仍必须继续检查。
- 启动初始化必须是幂等的，多次重启不能重复创建业务内容或破坏已有运行环境。
- 启动时不能创建任何具体引流工作目录，也不能生成默认产品、策略或其他业务数据。

相关参与模块会跨越启动入口、应用编排、存储适配器和共享错误处理，因此这是一个典型的跨模块基础能力变更。

## Goals / Non-Goals

**Goals:**
- 在后端启动阶段增加统一的运行时基础初始化流程。
- 启动时默认以当前工作目录下的 `cybernomads/` 作为运行根目录。
- 自动创建缺失的固定子目录：`product/`、`strategy/`、`work/`。
- 自动创建或打开 SQLite 文件。
- 建立运行时 SQL 资产加载与执行机制，并保证脚本可幂等执行。
- 仅创建系统级初始化表，用于记录脚本执行状态或版本信息。
- 当 SQLite 初始化或运行时 SQL 执行失败时，启动流程明确失败并返回可诊断错误。

**Non-Goals:**
- 不定义正式业务表结构。
- 不接入 `docs/sql/` 作为运行时 SQL 来源。
- 不创建任何具体引流工作目录，例如 `work/<具体工作>/`。
- 不生成默认产品 Markdown、策略 Markdown 或种子业务数据。
- 不引入完整的数据库迁移框架或版本升级工具链。

## Decisions

### 1. 使用“运行时 SQL 资产目录”作为 SQL 来源

**Decision**
- 启动初始化所需的 SQL 脚本由后端运行产物自带的运行时资产目录提供，例如 `runtime-assets/sql/`，而不是从 `docs/sql/` 读取。

**Rationale**
- `docs/sql/` 属于开发文档，不保证进入发布产物。
- 运行时初始化必须脱离文档目录独立存在，才能支持真实用户环境启动。
- 未来即使补全 `docs/sql/`，也可以通过构建时同步生成运行时 SQL 资产，而不改变启动机制。

**Alternatives Considered**
- 直接读取 `docs/sql/`
  - 放弃原因：运行时不可依赖开发文档目录。
- 在 TypeScript 代码中硬编码 SQL 字符串
  - 放弃原因：会让 SQL 难以维护，也不利于后续扩展多脚本执行。

### 2. 初始化流程只创建系统级初始化表，不创建正式业务表

**Decision**
- 本次启动初始化只执行系统级 SQL，例如创建脚本执行记录表，不创建产品、策略、任务、引流工作等正式业务表。

**Rationale**
- 当前业务表结构尚未定义，提前创建会制造伪设计。
- 先把启动机制和脚本执行机制打通，后续正式业务表只需按同一机制继续扩展。

**Alternatives Considered**
- 在本次提案中顺便定义首批业务表
  - 放弃原因：会扩大变更范围，也与当前需求目标不一致。

### 3. 目录存在时只跳过根目录创建，不跳过后续检查

**Decision**
- 如果当前目录下已存在 `cybernomads/`，初始化流程只跳过根目录创建动作，但必须继续检查固定子目录、SQLite 文件和运行时 SQL 脚本。

**Rationale**
- 仅凭根目录存在无法证明系统已经完整初始化。
- 这样可以避免“半初始化状态”被误认为可用，例如根目录存在但 SQLite 丢失或脚本从未执行。

**Alternatives Considered**
- `cybernomads/` 存在即视为已初始化并完全跳过
  - 放弃原因：风险过高，容易掩盖启动失败或目录不完整问题。

### 4. 启动初始化必须幂等且 fail-fast

**Decision**
- 初始化流程允许重复执行；已完成的目录创建与 SQL 脚本执行应被安全跳过。
- 一旦 SQLite 文件无法创建或运行时 SQL 执行失败，启动流程必须立即失败并返回明确错误。

**Rationale**
- 后端本地开发和用户实际使用都可能频繁重启，幂等是基础要求。
- fail-fast 能减少脏状态，避免系统以不完整环境继续运行。

**Alternatives Considered**
- 初始化失败时记录日志后继续启动
  - 放弃原因：会让系统带着不完整基础设施运行，后续故障更难排查。

### 5. 具体工作目录保持业务驱动生成

**Decision**
- 启动阶段只创建 `work/` 根目录，不创建任何 `work/<具体工作>/` 子目录。

**Rationale**
- 具体工作目录是引流工作域的业务对象，应在创建引流工作时生成。
- 如果启动时预创建，会混淆“运行时根环境”和“业务实例上下文”。

**Alternatives Considered**
- 启动时预创建默认工作模板目录
  - 放弃原因：会引入默认业务内容，超出本次变更范围。

## Risks / Trade-offs

- [运行时 SQL 资产与未来正式数据库设计可能分离] → 先将本次范围严格限定为系统初始化表，后续如果引入正式业务表，再建立文档到运行时资产的同步策略。
- [目录与 SQLite 初始化逻辑分散在多个模块中] → 通过应用层统一编排初始化入口，把磁盘与数据库操作收敛到存储适配层。
- [不同操作系统对路径、权限和大小写敏感性不同] → 所有路径与文件检查统一使用标准路径抽象与跨平台约束，不依赖平台特有行为。
- [重复执行 SQL 脚本可能造成状态污染] → 通过系统初始化表记录已执行脚本，保证脚本幂等执行。
- [当前只创建系统表，后续业务表扩展时可能需要额外迁移设计] → 接受该分阶段方案，把后续业务表设计作为独立变更推进。

## Migration Plan

1. 为后端增加运行时基础初始化入口，并在启动流程中接入。
2. 增加运行时 SQL 资产目录及首个系统初始化脚本。
3. 让启动流程在本地目录、SQLite 文件和脚本执行三个环节按顺序工作。
4. 通过本地重复启动验证幂等性，通过故障注入验证失败时的错误返回。
5. 如果初始化逻辑导致启动异常，可通过移除启动编排入口回退到当前空骨架状态，不影响现有业务能力，因为当前尚无正式业务实现依赖该机制。
