## Context

当前任务模块、SQLite 仓储、任务拆分 Skill、任务执行 Skill、API 文档和领域文档，都把 `input_needs_json` / `inputNeeds` 解释为“输入需求对象数组”。这一模型来源于早期为了让 Planner、后端和 Agent 共享结构化字段而做的抽象，但在真实业务中，任务输入并不是固定 schema 的结构体，而是一段由任务拆分 Agent 生成、由任务执行 Agent 消费的输入获取提示词。

这使当前实现出现了三个持续性偏差：

1. 任务拆分阶段被迫编造 `name/description/source` 三元组，无法自然表达复杂输入获取逻辑。
2. 任务执行阶段真正消费的是自然语言提示，而不是对象数组，因此 Skill 与后端契约存在语义错位。
3. SQLite 列名与 API 模型都在暗示 JSON 结构化数组，后续团队成员容易继续沿着错误方向扩展。

这次变更横跨任务领域类型、持久化映射、受控工具校验、任务拆分 Skill、任务执行 Skill、顶层 API/SQL 文档和 OpenSpec 主 specs，因此需要一份集中设计来约束迁移方向与兼容策略。

## Goals / Non-Goals

**Goals:**

- 将任务输入契约统一为“单段输入提示词”，并明确它的生产者是任务拆分 Agent，消费者是任务执行 Agent。
- 调整任务模块类型、持久化、读写 API、工具脚本和 Skill 文档，使它们对同一语义达成一致。
- 更新 `docs/api/tasks.yaml`、`docs/sql/tasks.sql`、领域设计文档与 OpenSpec 主 specs，避免顶层契约继续误导实现。
- 为现有已写入的历史数组数据提供清晰的迁移或兼容策略，避免运行时直接损坏已存在任务。

**Non-Goals:**

- 不在本次变更中设计新的任务调度算法、Planner 行为或 Agent Provider 协议。
- 不在本次变更中引入新的任务状态、任务产出 schema 或任务恢复机制。
- 不在本次变更中重构任务条件 `condition_json` 的建模方式。
- 不强制在本次第一阶段修改 SQLite 物理列名，只要语义能够先正确落地。

## Decisions

### 1. 将 `inputNeeds` 语义收敛为单字符串提示词

任务输入字段将从 `TaskInputNeed[]` 收敛为单个字符串字段，语义为“任务执行 Agent 的输入获取说明提示词”。

原因：

- 真实消费者是 Agent，不是静态规则引擎。
- 输入说明可能天然包含多步推理、相对路径提示、上游任务产出定位、失败时的处理方式，很难稳定压缩成固定对象数组。
- 这能让任务拆分 Skill 自然产出“可执行提示词”，减少伪结构。

备选方案：

- 保持对象数组并额外增加自由文本字段。未采用，因为会形成双真相来源，后续仍会出现 Agent 读提示词、后端维护数组的分裂。

### 2. 第一阶段保留 SQLite 列名 `input_needs_json`，但重定义其存储内容

本次实现阶段优先修改语义与读写逻辑，不立即要求物理列名重命名。

原因：

- 当前影响面已覆盖 API、Skill、工具脚本、测试和运行时，如果再叠加物理列改名，会提高迁移与回滚复杂度。
- 列名虽不理想，但短期可以通过 SQL 文档与 OpenSpec 说明明确“此列保存输入提示词文本”。

备选方案：

- 直接将列名改为 `input_prompt` 或 `input_prompt_text`。未作为第一阶段默认方案，因为这需要额外的数据迁移脚本与初始化表兼容处理，适合后续单独清理。

### 3. 运行时对旧数组数据提供兼容读取，但新写入统一为字符串

仓储层在读取 `input_needs_json` 时需要支持两类历史内容：

- 新模型：直接读取字符串提示词。
- 旧模型：如果内容是数组，则在读取时转换为一段兼容提示词，保证旧任务至少可被观察和执行。

原因：

- 当前本地环境和部分历史测试样本中已经存在数组结构。
- 直接硬切会让历史任务详情查询和执行 Skill 读取失败。

备选方案：

- 破坏式切换，不兼容旧数据。未采用，因为会让现有运行目录和历史集成验证失效，风险过高。

### 4. 任务拆分 Skill 与工具脚本改为“校验字符串存在”，不再校验数组项

任务拆分 Skill 输出契约、`task-set.js` 校验脚本、受控批量保存工具，都将只要求任务草稿提供非空输入提示词。

原因：

- 任务拆分阶段真正需要保证的是“执行 Agent 能看懂输入从哪里来”，而不是满足数组结构。
- 这能让受控工具继续承担边界校验职责，但校验目标回到真实业务语义。

备选方案：

- 让 Skill 继续输出结构化数组，再由后端拼装提示词。未采用，因为这样会把 Agent 语义翻译责任塞回后端，违背当前领域边界。

### 5. API 契约与领域文档同步改名或改注释，避免继续把它描述成集合

外部返回模型是否保留字段名 `inputNeeds`，需要结合前端兼容性决定；但无论是否保留名字，文档与规格必须明确它是“输入提示词”。

原因：

- 如果仅改实现不改契约说明，团队会继续误以为这是数组集合。
- 即使短期为兼容保留字段名，也要把描述从 “input needs collection” 改为 “execution input prompt”。

备选方案：

- 完全不改 API 名称与文案，只改内部实现。未采用，因为这会把错误语义固化到对外契约。

## Risks / Trade-offs

- [历史数据兼容复杂度] → 在仓储层增加读取兼容逻辑，并在测试中覆盖“旧数组 -> 新提示词”转换路径。
- [字段名与真实语义继续不一致] → 第一阶段允许保留列名，但必须在 SQL / API / OpenSpec 中明确语义，后续可再单独发起命名清理提案。
- [前端或调用方已依赖数组结构] → 在实现前先以 API 文档为准收口，必要时增加过渡说明，不在无通知情况下 silently 变更。
- [Skill 与运行时副本不一致] → 修改 `runtime-assets` 后，确保运行目录中的已安装 Skill 资产同步刷新或在实现中增加重新拷贝步骤。
- [Agent 生成提示词质量不稳定] → 在任务拆分 Skill 中补充输入提示词写作规则、自检要求和失败反馈要求。

## Migration Plan

1. 更新 OpenSpec delta specs，明确任务输入字段已从集合语义切换为输入提示词语义。
2. 更新顶层文档：
   - `docs/api/tasks.yaml`
   - `docs/sql/tasks.sql`
   - 任务领域设计文档
3. 修改任务模块类型、服务校验和 SQLite 仓储读写逻辑。
4. 修改任务拆分 Skill、任务执行 Skill、受控工具脚本与相关运行时资产。
5. 为旧数组数据增加读取兼容；新写入统一写入字符串。
6. 更新单元测试、集成测试和 Skill 资产测试。
7. 实施后通过任务拆分与任务详情查询链路验证输入提示词在创建、持久化、读取、消费四个阶段一致。

回滚策略：

- 若本次上线后出现执行 Skill 无法消费新字符串格式的问题，可先回滚到旧版本代码与 Skill 资产。
- 若已产生少量新字符串数据，旧版本读取会失败，因此回滚前应优先恢复兼容读取逻辑，或限制在单环境内验证后再扩大使用。

## Open Questions

- API 对外字段名是否仍保留 `inputNeeds`，还是同步更名为更明确的 `inputPrompt` / `executionInputPrompt`？
- SQLite 列名 `input_needs_json` 是否需要在后续第二阶段单独发起清理提案？
- 历史数组数据转换为兼容提示词时，采用通用拼接模板即可，还是需要在仓储层提供更强语义转换规则？
