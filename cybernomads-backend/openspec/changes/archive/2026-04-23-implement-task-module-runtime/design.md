## Context

`src/modules/tasks/` 当前为空，但任务领域需要作为引流工作任务拆分、线程规划器和 Agent 执行 Skill 之间的稳定协作点。任务模块不负责执行任务，也不负责调度算法；它负责保存和暴露任务、状态、条件、输入需求、任务集和产出记录。

## Goals / Non-Goals

**Goals:**
- 落地任务模块运行时。
- 支持任务读模型、状态回写和产出记录归档。
- 支持按引流工作批量创建任务集。
- 支持引流工作更新重建时替换当前任务集。
- 让 Agent 相关流程可以通过受控接口更新任务，而不是直接修改数据库。

**Non-Goals:**
- 不实现线程规划器。
- 不实现任务拆分 Agent Skill。
- 不实现 Agent provider 调用。
- 不定义任务产出数据本体 schema。
- 不实现自动重试、手动恢复、复杂状态机或任务历史复盘。

## Decisions

### 1. 模块围绕任务、任务集和产出记录实现

任务模块保存任务主记录，并保存任务产出记录。任务集不是额外聚合根，而是“同一引流工作下当前任务集合”的应用服务能力，用于承接 Agent 拆分结果。

### 2. 批量创建/替换是受控入口

引流工作创建后，应调用批量创建任务集能力；引流工作更新重建后，应调用替换当前任务集能力。替换必须由任务模块统一处理，不能由 Agent 删除文件、直接改库或绕过服务层完成。

### 3. 状态更新接口保持受控且简单

状态只允许使用 `ready / running / completed / failed`。Agent 执行 Skill 和规划器都必须通过模块接口更新状态，不能直接访问 SQLite。MVP 不强制实现复杂状态迁移图，但服务层应拒绝未知状态。

### 4. 条件与输入需求先按结构化 JSON 语义保存

任务拆分 Agent 需要灵活描述条件和输入需求，因此第一版保留结构化聚合表达，服务层只做最小校验，具体执行判断由规划器解释。

### 5. HTTP 接口服务普通 UI 和 Agent 工具调用

同一组任务接口既可用于前端执行台读取，也可作为 Agent 工具调用的受控后端入口。权限与鉴权策略当前不在 MVP 范围内；后续可拆分内部工具 API 与普通 UI API。

## Risks / Trade-offs

- [风险] 条件 JSON 过于自由，可能出现 Agent 生成格式不稳定。
  → 通过任务拆分 Skill 的输出规则和服务层最小校验降低风险。
- [风险] 状态更新接口若过宽，Agent 可能误改状态。
  → 明确只允许四态，并在 Skill 中强调完成/失败回写规则。
- [风险] 产出记录不存储数据本体，第一版查询能力有限。
  → 这是为保持任务域稳定边界而做的取舍。
- [风险] 替换任务集可能影响旧任务历史。
  → MVP 先只保证当前任务集可运行；历史归档后续独立提案。
