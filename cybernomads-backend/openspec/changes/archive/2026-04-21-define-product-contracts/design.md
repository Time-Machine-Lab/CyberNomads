## Context

Cybernomads 的产品领域边界已经在领域设计文档中被收敛为一个“薄领域、强输入”的模块：产品域负责定义产品对象、提供产品摘要、提供产品完整上下文，并支撑策略域与引流工作域理解“当前在推广什么”。但当前仓库中既没有产品模块的 API 文档，也没有产品模块的 SQL 文档，这与项目宪法中“`docs/api/` 与 `docs/sql/` 是单一真理”的要求冲突。

因此在进入产品模块实现前，需要先定义顶层契约，明确产品模块的外部行为、最小数据结构和一些已经确认的业务决策，包括：
- 产品名称不强制唯一
- 当前阶段不支持删除产品
- 产品列表返回摘要信息
- 产品详情返回完整 Markdown 正文
- “提供产品完整上下文”在当前阶段复用产品详情能力

## Goals / Non-Goals

**Goals:**
- 为产品模块建立首版 `docs/api/*.yaml` 契约。
- 为产品模块建立首版 `docs/sql/*.sql` 契约。
- 明确产品创建、更新、列表、详情的稳定业务边界。
- 固化名称不唯一、无删除、详情返回完整正文等核心决策。
- 为后续产品模块实现提案提供稳定输入边界。

**Non-Goals:**
- 不实现产品模块代码。
- 不引入产品删除行为。
- 不定义产品版本体系、草稿态、发布态、归档态。
- 不扩展到策略、引流工作、任务等其他领域的实现细节。
- 不新增独立的“产品上下文接口”，当前只定义复用详情能力的语义。

## Decisions

### 1. 先定义顶层 API 与 SQL 契约，再进入模块实现

**Decision**
- 本次变更优先产出 `docs/api/` 与 `docs/sql/` 中的产品模块契约文档，把产品模块实现留给下一份提案。

**Rationale**
- 这符合项目宪法，也能防止后续实现时边开发边猜契约。
- 产品模块后续会被策略域、引流工作域消费，先稳定契约可以减少返工。

**Alternatives Considered**
- 直接进入产品模块实现
  - 放弃原因：会违反当前项目的顶层治理要求。

### 2. 产品名称不作为唯一键

**Decision**
- 产品名称保持可读标识角色，不承担唯一性约束；唯一标识应由稳定 ID 承担。

**Rationale**
- 同名产品在真实使用中是可能出现的，例如不同阶段的相似推广对象。
- 如果把名称设计成唯一键，会把展示语义错误提升为业务主键语义。

**Alternatives Considered**
- 强制产品名称唯一
  - 放弃原因：约束过强，会把列表、创建、复制等场景变复杂。

### 3. 产品详情直接返回完整正文

**Decision**
- 对外产品详情接口直接返回完整 Markdown 正文，而不是只返回元数据或摘要。

**Rationale**
- 产品域的核心价值之一就是提供完整产品上下文。
- 这能减少后续调用方为了获取完整内容再发起额外请求。

**Alternatives Considered**
- 详情只返回元数据，另外提供单独 context 接口
  - 放弃原因：当前阶段会增加接口数量和调用复杂度，但业务价值不明显。

### 4. 产品列表与产品详情职责分离

**Decision**
- 产品列表用于选择和展示，只返回摘要信息；产品详情用于查看和消费，返回完整正文。

**Rationale**
- 列表与详情面向的使用场景不同，职责应分离。
- 这样也更符合“快速选择”与“完整语义消费”的不同需求。

**Alternatives Considered**
- 列表也返回完整正文
  - 放弃原因：返回内容过重，且不符合列表场景。

### 5. 当前阶段显式不支持删除产品

**Decision**
- 首版产品契约中不提供删除接口，也不定义删除相关存储语义。

**Rationale**
- 删除会立刻引入引用完整性、软删/硬删和清理语义问题。
- 当前阶段应先把创建、更新、查询闭环打通。

**Alternatives Considered**
- 同步设计删除能力
  - 放弃原因：会让契约边界变重，延缓产品模块首版落地。

## Risks / Trade-offs

- [先做契约、后做实现会增加一个前置步骤] → 接受这个阶段性成本，以换取后续实现更稳定的边界。
- [详情返回完整正文可能让接口响应更重] → 当前产品正文规模可控，优先保证调用方拿到完整语义；若后续正文明显变大，再独立拆 context 接口。
- [名称不唯一会增加列表选择歧义] → 通过稳定 ID 和摘要信息补充展示，不把名称误用为唯一键。
- [不支持删除可能让管理能力显得不完整] → 明确这是 MVP 范围约束，后续如有需要再单独扩展。

## Migration Plan

1. 在 `docs/sql/` 中创建产品模块首版 SQL 文档。
2. 在 `docs/api/` 中创建产品模块首版 API 文档。
3. 将产品领域设计中的关键决策映射到顶层契约文档。
4. 以本次契约提案为基础，再进入产品模块实现提案。

## Implementation Readiness

### Minimal Feature Checklist

- 创建产品：接收 `name` 与 `contentMarkdown`，生成稳定 `productId`，建立产品记录与正文引用关系。
- 更新产品：基于既有 `productId` 更新 `name` 与完整正文，保持同一产品对象语义不变。
- 查询产品列表：返回用于选择与展示的摘要视图，最小字段集合为 `productId`、`name`、`updatedAt`。
- 查询产品详情：返回 `productId`、`name`、`contentMarkdown`、`createdAt`、`updatedAt` 的完整详情视图。
- 提供完整产品上下文：当前阶段直接复用产品详情能力，不新增独立外部 context 接口。

### Entry Conditions For `implement-product-module`

- `docs/sql/products.sql` 已成为产品模块结构化存储的单一真理，字段边界已锁定为 `product_id`、`name`、`content_ref`、`created_at`、`updated_at`。
- `docs/api/products.yaml` 已成为产品模块 HTTP 契约的单一真理，接口边界已锁定为创建、更新、列表、详情。
- API 与 SQL 的命名映射已明确：`productId` ↔ `product_id`，`name` ↔ `name`，`createdAt` ↔ `created_at`，`updatedAt` ↔ `updated_at`，`contentMarkdown` ↔ 由 `content_ref` 关联装配的完整正文。
- MVP 范围已锁定不包含删除、草稿、发布、归档、审核、版本链和独立 context 接口，后续实现不得反向扩张契约。
- 后续实现提案可直接围绕“创建、更新、列表、详情、完整上下文提供”五项最小能力展开，无需再回头补定义顶层契约。
