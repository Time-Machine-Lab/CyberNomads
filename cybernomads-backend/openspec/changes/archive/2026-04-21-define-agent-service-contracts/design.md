## Context

Cybernomads 的 Agent 服务接入域已经在领域文档中明确了边界: 它负责管理系统当前连接的外部 Agent 服务、连接校验、当前激活服务以及能力准备状态，并向上层提供统一接入口。这个领域不负责任务规划语义、任务调度、subagent 内部生命周期，也不负责平台脚本执行细节。

当前仓库里还没有这个领域的顶层契约文档，而项目宪法要求 `docs/api/` 与 `docs/sql/` 成为接口和存储的单一真理。因此本提案先固定 Agent 服务接入域的外部行为边界和最小持久化语义，为后续运行时实现与 OpenClaw 适配提供稳定输入。

## Goals / Non-Goals

**Goals:**
- 为 Agent 服务接入域建立首批 `docs/api/*.yaml` 契约。
- 为 Agent 服务接入域建立首批 `docs/sql/*.sql` 契约。
- 固化单激活服务、连接校验、能力准备、状态查询等 MVP 边界。
- 确保顶层契约对 provider 保持中立，不把 OpenClaw 私有语义直接写进上层契约。
- 为后续 `implement-agent-service-access` 与 `implement-openclaw-adapter` 提供稳定前置条件。

**Non-Goals:**
- 不实现 Agent 服务接入模块代码。
- 不实现 OpenClaw 的具体网关调用。
- 不定义任务规划、任务执行调度或日志回写的上层业务契约。
- 不支持多 Agent 服务并行接入、故障切换、自动重试与容灾恢复。
- 不在本提案中解决 `docs/api/` 与 `docs/sql/` 最终归属目录的工程组织问题，只先固定契约内容要求。

## Decisions

### 1. 先定义顶层 API 与 SQL 契约, 再进入模块实现

**Decision**
- 本次变更优先产出 Agent 服务接入域的 `docs/api/` 与 `docs/sql/` 契约文档，运行时实现留给后续提案。

**Rationale**
- 这符合项目宪法，也能避免后续实现阶段边开发边猜接口和存储边界。
- Agent 接入域会被工作区、引流工作和任务相关上层模块复用，先稳定边界可以减少返工。

**Alternatives Considered**
- 直接进入运行时模块开发。
  - 放弃原因: 会违背当前仓库的文档先行原则。

### 2. 顶层契约围绕“单个当前激活 Agent 服务”建模

**Decision**
- 顶层契约按“系统当前只有一个激活中的外部 Agent 服务”设计，而不是多服务管理列表。

**Rationale**
- 这与当前 MVP 范围一致，也能显著降低前后端状态复杂度。
- 后续如果扩展为多服务模型，可以在不破坏 provider 抽象的前提下再演进。

**Alternatives Considered**
- 直接设计多服务并行接入契约。
  - 放弃原因: 超出 MVP，且会提前引入路由、优先级和切换语义。

### 3. 将“能力准备”作为业务状态表达, 而不是暴露安装动作

**Decision**
- 顶层契约表达“是否已完成 Cybernomads 所需能力准备”，而不是对上层暴露具体 Skill 安装动作细节。

**Rationale**
- 这更符合领域语义，也避免把 provider 内部操作方式固化成业务模型。
- 后续如果能力准备方式改变，上层契约仍可保持稳定。

**Alternatives Considered**
- 把 Skill 安装动作直接暴露为业务接口。
  - 放弃原因: 会让顶层契约被实现细节污染。

### 4. 顶层契约对 provider 保持中立

**Decision**
- API 和 SQL 契约中不引入 OpenClaw 私有命名、私有字段和私有生命周期语义。

**Rationale**
- Agent 接入域的价值之一就是解耦 provider 实现，顶层契约必须优先服务这个目标。

**Alternatives Considered**
- 先以 OpenClaw 字段形式建模, 后续再抽象。
  - 放弃原因: 会把 MVP 的实现选择错误固化为长期契约。

## Risks / Trade-offs

- [先做契约会增加一个前置阶段] -> 接受该阶段成本，以换取后续模块实现和 provider 适配更清晰的边界。
- [单服务模型会限制首期扩展性] -> 与 MVP 范围对齐，后续若进入多服务阶段再单独扩展契约。
- [能力准备语义较抽象] -> 通过 API 文档明确触发条件、状态字段和结果语义，避免实现时各自解释。
- [当前仓库未见 `docs/api/` 与 `docs/sql/` 目录] -> 在后续落地契约时先确认目录归属，但不影响本提案先固定行为边界。

## Migration Plan

1. 创建 Agent 服务接入域首批 SQL 契约文档。
2. 创建 Agent 服务接入域首批 API 契约文档。
3. 将领域文档中的关键边界映射到顶层契约, 固化单服务、连接状态与能力准备语义。
4. 以本次契约提案为基础，继续进入运行时实现与 OpenClaw 适配提案。

## Open Questions

- `docs/api/` 与 `docs/sql/` 的最终落地目录应位于仓库根目录还是后端子项目目录，需要在执行实现前做一次工程归属确认。
- 凭证在顶层 API 契约中是直接提交、引用已有安全存储，还是采用后续再补充的密文方案，当前仍需在实现前明确。
