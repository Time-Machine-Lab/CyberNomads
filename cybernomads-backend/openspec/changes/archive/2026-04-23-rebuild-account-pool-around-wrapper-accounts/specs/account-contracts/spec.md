## ADDED Requirements

### Requirement: Account contracts SHALL define wrapper-account identity separately from resolved platform identity
The system SHALL define top-level contracts around a stable wrapper account object identified by `accountId`, while treating resolved platform identity as optional profile data observed after validation rather than as the account's creation-time business key.

#### Scenario: Wrapper account identity does not require resolved platform UID at creation time
- **WHEN** account creation contracts are defined for the new model
- **THEN** the contracts SHALL require a stable `accountId` generated by the system and SHALL NOT require `resolvedPlatformAccountUid` during creation
- **AND** the contracts SHALL allow the resolved platform identity fields to be empty until a successful validation occurs

#### Scenario: Resolved platform identity may repeat across wrapper accounts
- **WHEN** top-level contracts define resolved platform profile fields
- **THEN** the contracts SHALL treat `resolvedPlatformAccountUid` as observed platform data instead of as a global uniqueness key
- **AND** the contracts SHALL allow multiple wrapper accounts on the same platform to carry the same resolved platform UID

### Requirement: Account contracts SHALL define internal profile fields separately from resolved platform profile fields
The system SHALL distinguish operator-managed account profile fields from platform-derived profile fields in top-level API and SQL contracts.

#### Scenario: Internal and resolved display names are stored separately
- **WHEN** account detail contracts are defined
- **THEN** the contracts SHALL include an operator-managed internal display name field and a separate resolved platform display name field
- **AND** the contracts SHALL NOT require successful validation to overwrite the internal display name

#### Scenario: Platform metadata and resolved profile metadata are isolated
- **WHEN** account detail contracts are defined
- **THEN** the contracts SHALL preserve operator-managed `platformMetadata` separately from resolved platform profile metadata
- **AND** platform profile refresh SHALL only update the resolved profile section

### Requirement: Account contracts SHALL publish account-bound connection-attempt contracts
The system SHALL publish dedicated connection-attempt API and SQL contracts for existing accounts, covering token input, QR challenge retrieval, validation, token application, and attempt logs.

#### Scenario: Connection-attempt API is bound to an existing account
- **WHEN** top-level API contracts define token connection workflows
- **THEN** the system SHALL publish a dedicated connection-attempt controller under the account resource
- **AND** the contracts SHALL NOT require a pre-account onboarding-session resource for new-account creation

#### Scenario: Connection-attempt SQL uses an independent process table
- **WHEN** top-level SQL contracts define token connection workflows
- **THEN** the system SHALL publish a dedicated `account_connection_attempts` table contract
- **AND** the main account table SHALL NOT embed the full connection-attempt state machine as inline columns

### Requirement: Account contracts SHALL define lifecycle, login, and availability status semantics
The system SHALL define independent lifecycle, login, and availability status dimensions together with explicit consumability semantics for wrapper accounts.

#### Scenario: New wrapper account starts as not logged in
- **WHEN** a new wrapper account is created without an active token
- **THEN** the contracts SHALL represent it with a first-class `not_logged_in` login status
- **AND** the account SHALL remain readable and editable as a valid account object

#### Scenario: Consumability depends on lifecycle, login, availability, and active token
- **WHEN** top-level contracts define internal token-consumption semantics
- **THEN** the contracts SHALL state that only accounts meeting the minimum consumability conditions can be treated as executable resources
- **AND** the contracts SHALL evaluate those conditions from lifecycle status, login status, availability status, and active-token presence

## REMOVED Requirements

### Requirement: Account contracts SHALL define a stable platform account identity model
**Reason**: The new model treats the account as a wrapper object owned by the system, not as a direct unique mapping of a platform-natural identity.
**Migration**: Replace creation-time `platformAccountUid` identity rules with `accountId`-based wrapper-account identity and move platform UID into resolved profile fields.

### Requirement: Account contracts SHALL separate active credentials from authorization attempts
**Reason**: The old authorization-attempt model is being replaced by a dedicated account-bound connection-attempt resource.
**Migration**: Move token-input, QR challenge, validation, and log semantics into the new connection-attempt contracts and keep active-token state on the account object.

### Requirement: Account contracts SHALL define status dimensions and minimum consumability semantics
**Reason**: The new product model replaces authorization-centric status language with login-centric status language.
**Migration**: Rewrite top-level status contracts around lifecycle, login, and availability semantics.

### Requirement: Account contracts SHALL publish token onboarding session contracts before stable account creation
**Reason**: The new model no longer uses a pre-account onboarding session to create accounts.
**Migration**: Create the wrapper account first, then use account-bound connection attempts for token access and validation.
