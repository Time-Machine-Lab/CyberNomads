## Context

产品领域设计已经明确了产品模块的核心职责：定义产品对象、提供产品摘要、提供完整产品上下文，并支持更新产品语义。与此同时，`initialize-runtime-foundation` 已经为后端提供了运行时目录和 SQLite 基础初始化能力，而 `define-product-contracts` 将负责补齐 `docs/api/` 与 `docs/sql/` 契约。

当前后端代码中，`src/modules/products/` 仍然是空骨架。产品模块的下一步不是继续讨论边界，而是把领域设计和顶层契约落成一个可运行的后端模块，至少完成以下闭环：
- 创建产品
- 更新产品
- 查询产品列表（摘要视图）
- 查询产品详情（完整正文视图）
- 向后续调用方提供完整产品上下文

这个实现会跨越领域对象、应用服务、HTTP Controller、SQLite 持久化和正文内容读写，因此需要单独一份实现设计。

## Goals / Non-Goals

**Goals:**
- 在后端实现产品模块的领域对象与应用服务。
- 在后端实现产品创建、更新、列表和详情查询能力。
- 让产品详情可直接承担“完整产品上下文”提供能力。
- 对齐 `docs/api/` 与 `docs/sql/` 产品契约后再进行具体实现。
- 对接当前运行时目录与 SQLite 基础能力。
- 为产品模块增加必要测试，覆盖核心主流程。

**Non-Goals:**
- 不实现产品删除。
- 不实现产品草稿/发布/归档/版本管理。
- 不扩展到策略、引流工作或任务模块的联动实现。
- 不在本次提案中抽象出通用 Prompt 资产平台。

## Decisions

### 1. 产品模块按“薄领域 + 明确应用服务”实现

**Decision**
- 产品模块保持轻量领域模型，在模块内围绕 `Product` 及其上下文提供能力组织代码，不额外引入复杂状态机。

**Rationale**
- 这与产品领域设计一致，也符合当前 MVP 的业务复杂度。
- 先把可运行闭环做扎实，比提前做过度抽象更重要。

**Alternatives Considered**
- 一开始就抽象成通用 Prompt 资产模块
  - 放弃原因：虽然产品和策略未来可能共享模式，但当前过早抽象会增加实现复杂度。

### 2. 产品详情复用为完整上下文能力

**Decision**
- 当前阶段不新增独立的外部“产品上下文接口”，而是让产品详情查询直接返回完整 Markdown 正文，承担完整上下文能力。

**Rationale**
- 这与第一份契约提案保持一致。
- 对调用方更直接，也减少额外接口和控制器复杂度。

**Alternatives Considered**
- 新增独立 context 接口
  - 放弃原因：当前阶段收益不足，且会增加接口面。

### 3. 列表与详情走不同返回模型

**Decision**
- 产品列表面向选择与展示，返回摘要模型；产品详情面向查看与消费，返回完整模型。

**Rationale**
- 能避免列表查询变成正文批量传输。
- 与策略绑定、引流工作选择等场景更匹配。

**Alternatives Considered**
- 列表与详情都返回完整正文
  - 放弃原因：不符合列表场景，也会增加不必要的数据负担。

### 4. 实现前必须先对齐顶层契约文档

**Decision**
- 在进入产品模块代码实现前，必须先校验或补齐 `docs/api/` 与 `docs/sql/` 产品契约，并以其为准。

**Rationale**
- 这是项目宪法的硬约束。
- 可以避免代码实现反向定义接口和表结构。

**Alternatives Considered**
- 先写代码，再补文档
  - 放弃原因：违反当前项目治理要求。

### 5. 产品模块依赖现有运行时基础能力，不重复定义启动责任

**Decision**
- 产品模块直接依赖当前已经存在的运行时基础能力，例如运行根目录与 SQLite 基础初始化，而不在产品模块内部重新承担这部分职责。

**Rationale**
- 避免产品模块越界到系统 bootstrap 职责。
- 让产品模块专注在业务能力本身。

**Alternatives Considered**
- 产品模块自行保证目录和数据库可用
  - 放弃原因：会和启动初始化能力边界重复。

## Risks / Trade-offs

- [产品模块实现依赖顶层契约先稳定] → 通过将 `docs/api/` 与 `docs/sql/` 校验列为实现前的首要任务降低风险。
- [详情返回完整正文可能让详情接口负载较重] → 当前产品正文规模可接受，优先满足语义完整性；后续若正文规模增长，再考虑拆分接口。
- [产品模块未来可能与策略模块形成重复模式] → 当前接受局部重复，待多个模块模式稳定后再考虑抽象。
- [如果存储实现边界不清，容易重新侵入领域层] → 在实现时将路径、落盘、数据库访问控制在适配层，保持产品模块的领域表达清晰。

## Migration Plan

1. 先完成或确认 `define-product-contracts` 中的 `docs/api/` 与 `docs/sql/` 文档。
2. 在 `src/modules/products/` 中建立产品模块实现骨架。
3. 对接 SQLite 与产品正文存取能力。
4. 实现 HTTP Controller 与应用服务闭环。
5. 增加测试，验证创建、更新、列表和详情行为。

## Open Questions

- 产品详情返回完整正文时，是否还需要补充额外的上下文元信息给后续消费方。
- 产品摘要模型最终是否需要包含更新时间等辅助字段，还是保持最小集合。
