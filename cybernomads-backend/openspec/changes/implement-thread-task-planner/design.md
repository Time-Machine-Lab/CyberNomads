## Context

架构文档已明确当前 MVP 采用线程决策，Agent 决策暂不实现。任务领域文档进一步说明，规划器会扫描任务，根据 `condition` 判断是否执行，并在提交给 Agent 服务后无需等待回复。

线程规划器处在应用层：它消费引流工作状态、任务条件和 Agent 服务统一入口，但不拥有这些领域模型本身。

## Goals / Non-Goals

**Goals:**
- 实现本地线程规划器的 MVP 闭环。
- 扫描 `running` 引流工作下的 `ready` 任务。
- 根据 `cron` 和 `rely_on` 评估是否提交执行。
- 提交 Agent 服务后继续轮询，不等待任务完成。
- 随应用生命周期启动和停止，避免测试或关闭时留下后台循环。

**Non-Goals:**
- 不实现 Agent 自主规划。
- 不实现失败重试、手动恢复、复杂容灾。
- 不实现多进程分布式调度或精确任务队列。
- 不实现平台脚本执行。
- 不负责创建或替换任务集。

## Decisions

### 1. 规划器属于应用层，不属于任务领域模型

任务域只提供任务条件和状态，规划器负责解释条件并提交执行。这样可以保留任务领域稳定性，也方便未来替换为 Agent 规划。

### 2. 采用 fire-and-forget 提交流程

当任务满足条件时，规划器先将任务标记为 `running`，再通过 Agent 服务统一入口提交任务执行请求。提交成功后不等待 Agent 回复，继续扫描下一项任务。

### 3. 提交失败不做重试，但需要显式落态

如果任务已被规划器选中但提交 Agent 服务失败，MVP 不做自动重试。实现应避免任务永久卡在不可观察的中间状态；可以将任务标记为 `failed` 并记录提交失败反馈，或在服务层返回明确错误。具体记录方式以任务模块现有能力为准。

### 4. rely_on 采用 MVP 更新时间判断

当前依赖判断按用户设想采用“依赖任务更新时间晚于当前任务更新时间”作为再次执行依据。该算法属于 planner 实现，未来可以扩展为更精细的依赖输出判断。

### 5. running 任务不重复提交

规划器只扫描 `ready` 任务，避免同一任务在 Agent 尚未回写完成前被重复分发。进程重启后已有 `running` 任务如何恢复不在本期自动处理。

### 6. 规划器必须可启动、可停止、可测试

规划器不应在模块 import 时自动启动。应用启动流程负责显式启动，应用关闭或测试 teardown 负责显式停止。

## Risks / Trade-offs

- [风险] 进程重启时 running 任务可能停留在运行态。
  → 当前 MVP 不做复杂恢复；后续可增加超时回收或人工处理。
- [风险] cron 表达式解析可能引入复杂依赖。
  → 第一版可选择轻量解析或最小可用表达，契约层不绑定具体库。
- [风险] fire-and-forget 会让提交成功与执行成功分离。
  → 这是当前设计目标，任务最终状态由 Agent 执行 Skill 回写。
- [风险] 1 秒扫描间隔在任务数量增长后可能浪费资源。
  → MVP 先保持简单，后续可配置化或升级为队列。

## Migration Plan

该 change 依赖任务模块运行时、引流工作运行时与 Agent 服务接入运行时。上线后规划器应随应用启动，并在应用关闭时停止轮询。

## Open Questions

- 第一版轮询间隔是否固定 1 秒；当前按架构设想采用 1 秒，后续可配置化。
