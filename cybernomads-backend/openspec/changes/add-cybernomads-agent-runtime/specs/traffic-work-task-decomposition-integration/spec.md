## ADDED Requirements

### Requirement: Traffic work decomposition SHALL stage Cybernomads Agent drafts before formal task persistence

The traffic work task decomposition integration SHALL stage task plan drafts generated by Cybernomads Agent and SHALL persist formal tasks only after Review and user confirmation.

#### Scenario: Create traffic work and generate draft

- **WHEN** a valid traffic work creation request is accepted
- **THEN** the backend SHALL prepare the traffic work context
- **AND** it SHALL request a structured task plan draft from Cybernomads Agent
- **AND** it SHALL save the draft as a decomposition artifact
- **AND** it SHALL NOT create formal task records before Review passes and the user confirms the plan

#### Scenario: Review passes and user confirms

- **WHEN** a task plan draft passes Agent Review and the user confirms the plan
- **THEN** the backend SHALL submit the confirmed snapshot through the task module
- **AND** it SHALL mark the traffic work context as prepared only after formal task persistence succeeds

#### Scenario: Review fails

- **WHEN** Agent Review fails or repair attempts are exhausted
- **THEN** the backend SHALL keep the traffic work outside prepared status
- **AND** it SHALL preserve the draft, Review issues, and failure reason for user inspection or feedback-based replanning

### Requirement: Traffic work decomposition SHALL support feedback-based replanning

The decomposition integration SHALL allow a new decomposition run to use prior artifacts, Review issues, execution failures, and user feedback as input.

#### Scenario: User requests replanning with feedback

- **WHEN** the user submits feedback for a task plan or task execution problem
- **THEN** the next decomposition run SHALL receive the prior draft, Review report, relevant execution failure summaries, and user feedback
- **AND** the new run SHALL record how feedback was considered in the resulting draft or report

#### Scenario: Execution failure becomes decomposition feedback

- **WHEN** OpenClaw execution of a confirmed task fails with a reason useful to planning
- **THEN** the system SHALL make that failure available to a future decomposition run
- **AND** the failure SHALL NOT cause OpenClaw to replan the entire traffic work by itself

### Requirement: Traffic work decomposition SHALL expose observable run progress

The decomposition integration SHALL expose enough run state for the frontend to show progress and explain what was produced.

#### Scenario: Read current decomposition run state

- **WHEN** the frontend reads a traffic work that is undergoing Cybernomads Agent decomposition
- **THEN** the backend SHALL expose the current stage, latest summary, Review status when available, and whether user confirmation is required
- **AND** it SHALL avoid exposing provider secrets or raw internal prompts as user-facing status

#### Scenario: Read decomposition report

- **WHEN** a decomposition run produces a report
- **THEN** the backend SHALL expose a readable summary of tasks, strategy coverage, dependencies, input sources, required resources, Review conclusion, and feedback options
