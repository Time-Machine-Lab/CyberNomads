## Context

产品模块当前是轻量 MVP 模块：产品元数据存储在 SQLite，产品正文以 Markdown 文件存储。产品身份由稳定 `productId` 承担，产品名称不唯一。当前已实现创建、更新、列表和详情读取，但删除接口被明确排除在早期 MVP 外。

现在需要补充删除能力，但仍保持产品模块边界简单：只删除产品自身数据，不处理引用关系。

## Goals / Non-Goals

**Goals:**
- 新增产品删除 API。
- 删除产品元数据。
- 删除产品正文 Markdown 文件。
- 删除不存在产品时返回明确错误。
- 删除后产品不再出现在列表中，详情读取返回 not found。

**Non-Goals:**
- 不检查产品是否被引流工作、策略、任务上下文或历史记录引用。
- 不实现软删除、归档、回收站或恢复。
- 不实现批量删除。
- 不删除已经生成到引流工作上下文中的产品快照。
- 不修改引流工作、任务、策略等其他领域语义。

## Decisions

### 1. 删除是硬删除

本次删除语义是移除产品元数据和产品正文文件，不保留 `deletedAt`、`archivedAt` 或产品状态字段。

### 2. 删除前先读取产品记录

删除需要先读取产品元数据，以获取 `contentRef`。若产品不存在，直接返回 not found，不尝试删除文件。

### 3. 文件删除与元数据删除保持产品模块内一致

产品删除应同时删除 SQLite 元数据和 Markdown 文件。实现时需要考虑失败顺序：

- 若文件删除失败，应避免数据库已经删除但文件残留且调用方无法感知。
- 若数据库删除失败，应避免文件已经删除但产品详情仍指向缺失文件。

推荐实现顺序是先确认产品存在，再删除文件，最后删除元数据；如果文件不存在但产品元数据存在，可以视为需要清理的残留状态，删除元数据并返回成功或按现有文件存储语义决定。具体实现需配合当前文件存储适配器行为。

### 4. 不做引用保护

即使产品已被引流工作引用，也允许删除。后续读取已有引流工作时如何展示缺失产品引用，属于引流工作模块的引用容错问题，不在本 change 处理。

## Risks / Trade-offs

- [风险] 删除被引流工作引用的产品后，旧引流工作详情可能只能显示 productId 或降级信息。
  → 用户已明确本次不考虑引用，后续如需要引用保护或软删除再单独设计。
- [风险] 硬删除不可恢复。
  → MVP 简化，避免引入状态机和回收站。
- [风险] 文件与数据库删除存在部分失败。
  → 实现和测试需要覆盖文件删除失败、产品不存在等情况，让失败可见。

## Migration Plan

无需数据迁移。现有产品记录继续有效；新增删除接口后可以删除已有记录及其正文文件。
