## Context

任务领域文档已经把任务定义为“由 Agent 基于引流工作拆分出的原子工作单元”，并明确了 `ready / running / completed / failed` 的轻量状态机、`cron / rely_on` 执行条件、`input_need` 输入需求和任务产出记录。

当前代码层面 `src/modules/tasks/` 仍为空，`docs/api/` 和 `docs/sql/` 也没有任务契约。除此之外，引流工作创建/更新后需要 Agent 拆分任务并形成任务集，因此契约必须同时覆盖单任务读写和任务集批量创建/替换。

## Goals / Non-Goals

**Goals:**
- 定义任务域的 API、SQL 和主规格。
- 固定任务归属、状态、执行条件、输入需求和产出记录的最小契约。
- 定义按引流工作批量创建/替换任务集的受控契约。
- 为 Agent 任务拆分、任务执行 Skill、线程规划器、任务模块运行时和引流工作任务拆分集成提供统一对齐点。

**Non-Goals:**
- 不实现业务代码。
- 不定义任务调度线程、轮询算法或并发策略。
- 不定义产出数据本体结构，例如视频、评论、私信、图片或文章的内部 schema。
- 不引入复杂状态机、重试、手动恢复或取消任务能力。
- 不定义旧任务历史归档、复盘或版本对比能力。

## Decisions

### 1. 将任务状态固定为四态

契约只定义 `ready`、`running`、`completed`、`failed`。`ready` 表示任务可被规划器纳入判断，但不等于一定可执行；真正是否执行由条件评估决定。这样能降低 Agent 执行 Skill 回写状态时的复杂度。

### 2. 将执行条件作为结构化声明，但不把调度算法写入契约

契约表达 `cron` 与 `rely_on` 两类条件，但不规定具体轮询线程如何计算下一次执行时间，也不规定依赖更新时间比较算法。算法属于规划器实现，条件声明属于任务域契约。

### 3. 将任务产出拆为产出记录和产出数据本体

任务产出记录只承载追踪语义，例如记录 ID、任务 ID、产出描述、数据位置、创建时间。真实产出数据留在任务数据存储区域或其他产物位置中，不在 SQL 契约里枚举类型字段。

### 4. 将任务与引流工作建立强归属关系

任务必须归属于引流工作，并通过 `trafficWorkId` 建立稳定引用。任务契约不单独定义引流工作更新后的业务触发条件，该语义继续服从引流工作域。

### 5. 将任务集创建/替换作为受控内部能力

任务通常不是用户逐条手工创建，而是 Agent 基于引流工作拆分后形成任务集。契约需要支持“为某个引流工作创建任务集”和“为某个引流工作替换当前任务集”两类受控能力。该能力可以暴露为内部 API、应用服务端口或 Agent 可调用工具，但必须经过任务模块校验与持久化，不能让 Agent 直接写数据库。

### 6. 替换任务集不等于定义历史归档策略

引流工作更新重建时可能需要替换任务集。当前契约只表达“当前可运行任务集”的替换语义，不在本 change 中定义旧任务历史如何归档、如何恢复或如何展示。若后续要做复盘或历史对比，应单独扩展任务历史或运行记录能力。

## Risks / Trade-offs

- [风险] `input_need` 和 `condition` 以聚合结构表达，未来可能需要更强类型化。
  → 先保持可扩展结构，避免在任务拆分 Skill 尚未稳定时过早固定字段。
- [风险] 产出记录过于抽象，前端第一版只能展示描述和位置。
  → 这是有意取舍，避免任务域提前变成日志系统或产物 schema 仓库。
- [风险] 四态状态机无法表达阻塞、取消、重试中等复杂状态。
  → 当前 MVP 明确避免复杂状态，后续如需要再通过新 change 扩展。
- [风险] 替换任务集可能让旧任务的历史语义变弱。
  → MVP 先保证当前引流工作可运行；历史保留与复盘能力后续独立设计。

## Migration Plan

本 change 只新增顶层契约，不执行数据迁移。后续运行时实现必须先同步 runtime SQL 脚本，再按该契约实现模块。

## Open Questions

- `cron` 是否在 API 契约层使用标准 cron 字符串，还是允许后续扩展为 interval 简化表达；本次先按 cron 语义描述，不额外扩展。
