# 任务拆分方法

## 核心目标

把一份引流工作拆成一组后续 subagent 可以独立执行的任务，而不是拆成一组抽象口号。

## 拆分顺序

1. 先从策略中提取“必须完成的阶段”。
2. 再从每个阶段中提取“可以独立交给一个 subagent 的原子任务”。
3. 再判断每个任务需要什么输入、依赖谁、产出什么、多久执行一次。
4. 最后再决定哪些资源要复制、哪些共享知识要准备、哪些数据文件要预留。

## 原子化判断

一个任务通常应该同时满足：

- 只有一个清晰目标
- 可以明确说明输入来源
- 可以明确说明主要产出
- 不需要无限扩张上下文才能执行

优先拆成这类任务：

- 搜索与收集
- 筛选与评估
- 草拟与生成
- 发布与发送
- 观察与跟进

避免拆成这类任务：

- “负责整套广撒网策略”
- “完成所有平台宣传”
- “处理所有后续互动”

## 条件设计

- `cron`：用于表达这个任务多久应该被重新考虑执行一次。
- `relyOnTaskKeys`：用于表达这个任务依赖哪些上游任务先完成或先产生产出。

常见判断：

- 搜索类任务通常适合声明 `cron`
- 跟进类任务通常同时需要 `cron` 和上游依赖
- 消费上游结果的任务至少要声明依赖关系

## 对象绑定使用

- 如果策略中存在账号、素材、图片等对象绑定，任务说明里要明确这些对象在什么时候使用。
- 不要只写“使用合适账号”；要结合对象绑定说明实际用哪个槽位资源。

## 任务说明应该写什么

每个任务的 `instruction` 至少应说明：

- 任务目标
- 执行步骤
- 需要读取哪些上下文、Skill、Knowledge 或工具
- 预期产出是什么
- 什么时候算完成

## 不要做什么

- 不要在这个阶段设计线程规划器实现。
- 不要写 provider 会话协议。
- 不要把平台自动化底层实现塞进任务说明。
- 不要让某个任务承担整份策略的全部上下文压力。
