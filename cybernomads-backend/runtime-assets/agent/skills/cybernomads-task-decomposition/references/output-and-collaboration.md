# 数据产出与任务协作

## 任务产出怎么想

任务产出分两层：

- 真实数据层：保存在当前引流工作 `data/` 区域里的文件或目录
- 抽象记录层：后续通过 Cybernomads 受控接口登记的产出记录

任务拆分阶段至少要把真实数据层规划清楚，让后续执行 Agent 知道数据应该存哪里、下游又该从哪里取。

## 数据规划原则

- 优先让一个任务拥有自己的数据文件，降低并发覆盖风险。
- 能用 `json` 表达的结构化数据，优先使用单任务单文件方式。
- 如果某个任务会产出图片、文章、评论草稿等多种文件，也要在任务说明中明确各自产物放置位置。

## 协作表达方式

当任务 B 需要消费任务 A 的产出时，同时做好两件事：

1. 在 `condition.relyOnTaskKeys` 中声明 B 依赖 A
2. 在 B 的 `inputPrompt` 中写明要从 A 的什么产物位置或什么数据文件读取，以及如何消费这些数据

## instruction 里要写明什么

任务说明中应明确：

- 数据最终存到 `data/` 下哪个文件或子目录
- 下游任务如何定位这些数据
- 产出记录将来应当怎样描述这份数据

## 推荐写法

- 候选视频类任务：`data/search-candidate-videos.json`
- 评论草稿类任务：`data/comment-on-prospects.json`
- 私信跟进类任务：`data/private-message-follow-up.json`

以上只是推荐模式，不是唯一固定格式。关键是：

- 同一任务的数据位置要稳定
- 下游任务可以据此消费
- 不要让所有任务共写一个含义不清的大文件
