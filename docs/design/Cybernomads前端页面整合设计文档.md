# CyberNomads 前端页面整合设计文档

## 1. 背景与目标

当前 UI 设计稿来自一组分散页面，其中既包含一级入口页，也包含编辑页、详情页、工作区内部子页和空状态页。如果直接按文件夹逐张实现，会出现以下问题：

- 一级导航被错误放大，导致结构混乱
- 编辑页、详情页和平时不常驻的内部页被误做成主模块
- 同一业务对象在不同页面之间的数据表达不一致
- Mock 数据会退化成按截图拼装的孤立假数据，后续无法接真实接口

本设计文档的目标是把现有设计稿整合成一套正常产品结构，并为后续前端实现提供统一的页面层级、路由规划、交互主链路和 Mock 数据模型。

## 2. 设计稿盘点与分类

### 2.1 分类原则

- 出现列表、创建按钮、卡片集合、筛选区的页面，优先判断为一级列表页
- 出现保存、部署、提交、连接测试等操作且围绕单个对象展开的页面，判断为创建页、编辑页或详情配置页
- 出现 breadcrumb、父实体上下文或局部标签切换的页面，优先判断为父模块内部子页
- 出现“未检测到”“立即配置”“初始化”等提示的页面，优先判断为空状态页，不独立建主导航

## 3. 统一信息架构

根据当前设计稿，一级导航统一收敛为五个主模块：

- 工作区
- 资产
- 策略
- 账号
- Agent

其余页面一律视为这五个模块内部的子页、配置页或空状态页。

```text
CyberNomads
├── 工作区
│   ├── 工作区列表
│   ├── 创建工作区
│   ├── 工作区详情 / 执行台
│   └── 任务干预页
├── 资产
│   ├── 资产列表
│   └── 资产编辑器
├── 策略
│   ├── 策略库
│   └── 策略编辑器
├── 账号
│   ├── 账号池列表
│   └── 账号详情与认证中心
└── Agent
    ├── Agent 总览
    ├── Agent 空状态
    └── OpenClaw 配置页
```

## 4. 路由规划

建议采用以下路由结构：

```text
/workspaces
/workspaces/new
/workspaces/:workspaceId
/workspaces/:workspaceId/execution
/workspaces/:workspaceId/tasks/:taskId/intervention

/assets
/assets/new
/assets/:assetId/edit

/strategies
/strategies/new
/strategies/:strategyId/edit

/accounts
/accounts/:accountId

/agents
/agents/openclaw
```

路由约束如下：

- `/agents` 同时承载正常总览态和空状态态，禁止额外拆出 `/agents/empty`
- `/workspaces/:workspaceId` 或 `/workspaces/:workspaceId/execution` 才是执行台承载页，禁止把 `_6` 单独提升为一级模块
- `/workspaces/:workspaceId/tasks/:taskId/intervention` 只从执行台进入，不能从一级导航直达
- 资产、策略、账号的编辑与详情页必须从各自列表页进入

## 5. 页面整合策略

### 5.1 工作区模块

工作区模块是当前产品主入口，负责把“资产 + 策略 + 账号”组合成真正可启动的执行单元。

- 使用 `_11` 作为工作区列表页基础
- 使用 `_4` 作为创建工作区流程页
- 使用 `_6` 作为工作区详情中的执行台主视图
- 使用 `_1` 作为任务干预页

工作区模块内部推荐交互链路：

```text
工作区列表
  -> 点击新建
  -> 选择资产 / 策略 / 账号
  -> 创建完成
  -> 进入工作区详情执行台
  -> 查看任务图与运行日志
  -> 点击某个任务
  -> 进入干预页发送人工指令
```

### 5.2 资产模块

资产模块负责管理面向引流执行的内容资产。

- 使用 `_7` 作为资产列表页
- 使用 `_2` 作为资产创建与编辑页

整合原则：

- 新建和编辑可以共用同一页壳，只按模式切换
- 编辑器以 Markdown 为主，不在 MVP 阶段引入复杂富文本能力

### 5.3 策略模块

策略模块负责策略内容与模块化策略配置。

- 使用 `_10` 作为策略库列表页
- 使用 `_5` 作为策略编辑器

整合原则：

- 列表态与编辑态分离
- 保存草稿和部署策略属于同一编辑上下文中的两个动作，不拆成不同页面

### 5.4 账号模块

账号模块负责平台账号接入、认证和状态查看。

- 使用 `_8` 作为账号池列表页
- 使用 `_3` 作为账号详情与认证中心

整合原则：

- 列表页只负责管理入口和状态摘要
- 具体凭证、认证、连接校验都放在详情配置页完成

### 5.5 Agent 模块

Agent 模块负责节点接入、状态查看与配置维护。

- 使用 `agent` 作为 Agent 总览页
- 使用 `_9` 作为 `/agents` 中的空状态视图
- 使用 `agent_openclaw` 作为 OpenClaw 配置页

整合原则：

- `_9` 不独立成一级页面，而是 `/agents` 的首启态
- 当系统尚未检测到可用代理节点时，`/agents` 渲染空状态和推荐配置入口
- 当节点已初始化后，`/agents` 渲染正常总览页

## 6. 主交互链路

基于 MVP 目标，前端必须先覆盖以下主链路：

### 6.1 首次进入产品链路

```text
进入工作区列表
  -> 发现缺少 Agent / 账号等基础资源
  -> 跳转 Agent 模块完成初始化
  -> 跳转账号模块补齐账号资源
  -> 返回工作区列表继续创建
```

### 6.2 基础资源准备链路

```text
资产列表 -> 新建资产 -> 保存
策略库 -> 新建策略 -> 保存草稿 / 部署
账号池 -> 新增账号 -> 完成认证
Agent -> 初始化节点 -> 进入配置页 -> 保存并返回总览
```

### 6.3 工作区执行链路

```text
工作区列表
  -> 新建工作区
  -> 绑定资产 / 策略 / 账号
  -> 进入执行台
  -> 查看任务图
  -> 查看实时日志
  -> 进入任务干预页
  -> 发送干预命令
```

## 7. Mock 数据设计

Mock 数据必须按领域实体组织，而不是按页面拆分。推荐先沉淀以下七类核心模型：

| 模型 | 作用 | 主要使用页面 |
|---|---|---|
| `workspace` | 工作区基础信息、绑定关系、运行状态摘要 | 工作区列表、创建页、执行台 |
| `asset` | 资产列表项、资产编辑内容、状态摘要 | 资产列表、资产编辑器 |
| `strategy` | 策略列表项、策略正文、模块配置、部署状态 | 策略库、策略编辑器 |
| `account` | 平台账号、认证状态、可用性状态 | 账号池、账号详情 |
| `agentNode` | Agent 节点信息、连接状态、能力状态 | Agent 总览、Agent 空状态、配置页 |
| `taskRun` | 任务节点、运行状态、时间信息、日志索引 | 执行台、任务图、运行日志 |
| `interventionRecord` | 人工干预记录、Agent 回复、时间线 | 任务干预页 |

Mock 场景必须至少覆盖以下状态：

- 空状态：未初始化 Agent、暂无工作区、暂无资产、暂无策略
- 列表基础态：存在若干列表项且可进入详情
- 编辑态：表单已有回填数据，支持保存、草稿、部署、提交
- 运行态：工作区执行中，任务图和日志持续变化
- 异常态：请求失败、认证失败、日志拉取失败、连接测试失败

Mock API 组织建议：

- 统一通过 `shared/api` 暴露接口入口
- mock adapter 负责根据环境切换到本地数据实现
- 页面层只消费统一接口，不直接 import 假数据文件
- 任务日志和运行状态变化可以通过定时器或轮询响应模拟，不在页面层自行拼装

## 8. 实现顺序建议

建议按以下顺序推进前端实现：

1. 先搭建应用壳、主导航和路由骨架，只建立五个主模块与必要子路由
2. 先实现统一 mock 模型与 mock API 入口，再落各页面壳
3. 优先完成工作区、Agent、账号三条主链路，因为它们决定产品能否形成完整可走通路径
4. 再补资产与策略的编辑器页面
5. 最后补执行台中的动态细节、任务干预页和异常态

## 9. 结论

当前这批 UI 设计稿已经足够覆盖 CyberNomads MVP 的主要前端结构，但前提是必须先完成页面整合，而不是按设计稿目录逐张实现。整合后的产品结构应以五个一级模块为核心，其余页面都作为模块内部子视图来实现。只有在这个前提下，路由设计、组件复用、Mock 数据和后续真实接口接入才会保持一致。
