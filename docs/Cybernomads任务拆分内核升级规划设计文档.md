# Cybernomads - 任务拆分内核升级规划

## 1. 本期目标与定位

### 1.1 本期愿景陈述

本期结束后，用户在创建或更新引流工作时，不再面对一条黑盒式任务拆分链路，而是可以由 Cybernomads 自有 Agent 生成、审查、修正并提交一份可理解、可反馈、可落地的任务方案，OpenClaw 则专注承担后续单任务执行。

### 1.2 与项目总方向的对齐

本期直接服务于 Cybernomads 项目总方向中“把增长策略变成 AI 可以理解、拆解和执行的任务链路”这一核心价值主张。相比第一期 MVP 已经打通的“产品 + 策略 + 引流工作 + Agent 执行”基础链路，本期重点把任务拆分从外部 Agent 自由执行动作升级为 Cybernomads 自己掌控的任务生成与质量判断能力。

### 1.3 本期核心要解决的问题

- 问题 1：当前任务拆分主要依赖外部 Agent 自行遵守流程，用户无法判断拆分是否合理，也难以及时发现任务过粗、过碎、依赖错误、材料缺失等问题。  
  期望：用户可以看到一份经过 Cybernomads Agent Review 的任务方案，并能理解每个任务为什么存在、如何协作、需要什么材料、风险在哪里。
- 问题 2：当前任务拆分结果一旦进入落库和运行，重新拆分缺少反馈闭环，用户的运行问题和人工判断无法稳定进入下一次拆分。  
  期望：用户可以基于本次任务拆分结果、Review 结论和运行异常反馈发起重拆，让下一次方案不是重新抽取，而是有依据地修正。

## 2. 功能模块规划

### 2.1 Cybernomads Agent 任务拆分中心

**模块一句话价值**：让 Cybernomads 自己掌握任务拆分的核心判断权，而不是把任务结构完全交给外部执行 Agent 自由生成。

#### 2.1.1 背景与目的

用户需要的是可执行的增长任务链路，而不是一组看似完整但质量不可控的自然语言任务。该模块的目的是让 Cybernomads 自有 Agent 负责理解产品、策略和对象绑定，并生成一份可审查的任务方案。

#### 2.1.2 核心功能描述

1. 用户创建或更新引流工作后，进入任务拆分准备阶段。
2. 用户看到 Cybernomads Agent 正在理解产品、策略和对象绑定。
3. 用户等待 Agent 生成一份任务方案草案，而不是立即落库任务。
4. 用户在拆分完成后查看任务列表、任务目标和任务之间的协作关系。
5. 用户可以继续确认、反馈或发起重新拆分。

#### 2.1.3 预期交付形态

任务拆分中心呈现为引流工作详情页中的一个独立区域。用户进入后可以看到“正在理解上下文”“正在生成任务方案”“等待 Review”“可确认”等阶段状态。任务方案生成后，页面以任务图和任务摘要列表展示每个任务的目标、输入、输出、依赖和所需资源，让用户先审阅方案，再决定是否进入运行准备。

#### 2.1.4 本期不做（模块级边界）

- 不自研完整通用 Agent 平台
- 不替代 OpenClaw 的任务执行能力
- 不支持多个拆分 Agent 并行竞争
- 不做跨项目任务模板市场

### 2.2 Agent Review 质量门禁

**模块一句话价值**：让任务拆分结果先经过智能审查和修正，再成为可运行任务。

#### 2.2.1 背景与目的

任务拆分差时，单纯增加进度面板并不能提升质量。该模块的目的是将“后端门禁”从硬编码规则转为 Cybernomads Agent Review，让审查者基于任务目标、依赖、输入输出、工具材料和运行可行性判断方案是否可用。

#### 2.2.2 核心功能描述

1. 用户看到任务方案进入 Review 阶段。
2. Agent Review 检查任务是否过大、过小、重复或缺少明确产出。
3. Agent Review 检查任务之间的依赖关系和数据协作是否合理。
4. Agent Review 检查任务所需工具、知识和材料是否已经被声明。
5. Review 不通过时，用户看到问题报告和重新生成结果。
6. Review 通过后，用户看到一份可确认的拆分报告。

#### 2.2.3 预期交付形态

Review 结果以“通过 / 需修正 / 失败”三类状态展示。通过时，页面展示任务质量摘要；需修正时，页面展示问题清单，例如“任务目标过大”“下游输入来源不清”“缺少执行材料说明”“任务产出不可被后续消费”。用户不需要理解底层校验细节，只需要知道当前任务方案是否可信，以及哪里需要修正。

#### 2.2.4 本期不做（模块级边界）

- 不引入硬编码质量评分系统
- 不要求 Review 给出百分制分数
- 不承诺完全自动判断业务优劣
- 不做真实平台动作试运行

### 2.3 任务拆分报告与用户反馈入口

**模块一句话价值**：让用户知道本次到底拆了什么、为什么这么拆，以及如何把问题反馈给下一次重拆。

#### 2.3.1 背景与目的

用户不只是想看到任务存在，更想知道任务是否服务于自己的增长目标。该模块的目的是把任务拆分结果转化成一份可读报告，并让用户能针对报告和运行问题提供反馈。

#### 2.3.2 核心功能描述

1. 用户查看本次任务拆分报告。
2. 用户看到每个任务对应的策略目标和业务价值。
3. 用户看到任务之间的数据流和协作关系。
4. 用户看到所需工具、材料和知识文件的准备情况。
5. 用户可以选择“确认使用”或“反馈后重拆”。
6. 用户可以选择反馈类型，例如任务太粗、依赖错误、缺工具、产出不清楚、策略理解错误。

#### 2.3.3 预期交付形态

拆分报告位于任务图旁边，采用摘要卡片和问题清单组合呈现。顶部说明本次拆分覆盖了哪些策略目标，中部展示任务协作路径，底部展示 Review 结论和用户反馈入口。用户可以针对整份方案反馈，也可以针对单个任务反馈，让下一次重拆带着明确问题继续推进。

#### 2.3.4 本期不做（模块级边界）

- 不做复杂协同评论系统
- 不做多人审批流
- 不做历史报告全文检索
- 不做自动增长效果归因

### 2.4 OpenClaw 执行者边界重塑

**模块一句话价值**：让 OpenClaw 专注执行已确认的单个任务，避免它继续承担 Cybernomads 的任务规划质量责任。

#### 2.4.1 背景与目的

OpenClaw 适合作为任务执行者和外部 Agent 能力入口，但任务拆分质量应该由 Cybernomads 自己负责。该模块的目的是重新划清职责：Cybernomads Agent 负责拆分和 Review，OpenClaw 负责执行已经确认的任务。

#### 2.4.2 核心功能描述

1. 用户确认任务方案后，引流工作进入可运行准备状态。
2. 用户启动引流工作后，系统继续按任务条件选择待执行任务。
3. OpenClaw 接收单个任务的执行说明和上下文。
4. OpenClaw 执行任务后回写状态、产出和异常信息。
5. 用户在任务执行不正常时，可以把运行问题反馈回任务拆分报告。

#### 2.4.3 预期交付形态

用户在执行台看到的仍然是任务图、状态和产出记录，但任务执行阶段不再混杂“重新规划整份引流工作”的行为。每个任务的执行说明来自已确认的 Cybernomads 任务方案，OpenClaw 只围绕当前任务执行，并把结果反馈到 Cybernomads 的任务记录和后续重拆入口。

#### 2.4.4 本期不做（模块级边界）

- 不取消 OpenClaw 接入
- 不重写现有任务调度器
- 不让 OpenClaw 负责整份任务重规划
- 不实现多执行 Agent 路由

## 3. 本期整体交付形态

本期完成后，用户在工作区创建引流工作时，会先进入一个可观察的任务拆分流程。页面显示 Cybernomads Agent 正在读取产品、策略和对象绑定，并逐步生成任务方案。方案生成后，用户不会立即看到一组已经落库且难以修改的任务，而是先看到任务图、任务摘要和拆分报告。报告中说明每个任务服务哪个策略目标、需要什么输入、会产出什么结果、与上下游任务如何协作，以及 Review 发现过哪些问题。

当 Review 通过后，用户可以确认使用该方案，引流工作才进入可启动状态。若用户认为任务拆得不合理，可以在报告中选择问题类型并补充说明，系统带着这些反馈重新生成和 Review。引流工作启动后，OpenClaw 只负责执行已经确认的单个任务；如果执行异常，用户可以把异常反馈回本次拆分报告，作为下一次重拆的依据。

## 4. 本期成功标准

| 验收维度 | 成功条件 | 验证方式 |
| :--- | :--- | :--- |
| 任务拆分主导权 | 新建或更新引流工作后，任务拆分由 Cybernomads Agent 生成任务方案草案，而不是由 OpenClaw 直接自由落库 | 产品走查，创建一次引流工作并检查拆分流程 |
| Review 质量门禁 | 每次任务方案确认前，都能看到 Agent Review 的通过状态或问题报告 | QA 执行通过与不通过两类样例 |
| 拆分结果可理解 | 用户能在报告中看到任务目标、输入、输出、依赖和所需资源说明 | 产品走查，使用一份真实产品和策略样例验证 |
| 用户反馈重拆 | 用户可以针对拆分结果提交反馈，并基于反馈重新生成任务方案 | QA 执行“反馈后重拆”流程 |
| OpenClaw 职责收敛 | 启动引流工作后，OpenClaw 只接收单任务执行请求，不再负责整份任务拆分落库 | 执行日志与产品走查 |
| 运行问题可回流 | 单任务运行异常后，用户可以将异常原因记录为下一次任务重拆的反馈来源 | 任务执行异常样例验证 |
| 用户体验完整性 | 一个熟悉产品概念但不了解实现细节的用户，可以理解本次拆分是否可用，并完成确认或反馈 | 内部可用性测试 |

## 5. 参考约束与衔接说明

### 5.1 上期回顾

上一阶段已经完成了引流工作创建、任务拆分 Skill、任务保存、任务执行调度和 OpenClaw 接入的基础链路。现有链路证明了“产品 + 策略 + 引流工作”可以被转化为任务并进入运行，但任务拆分质量、进度可见性、用户反馈和重拆闭环仍然不足。

### 5.2 下期预告

下期可以继续围绕以下方向展开：

- 将高质量任务拆分沉淀为可复用的增长任务模式。
- 引入更细的运行反馈归因，让失败原因更准确地回流到拆分阶段。
- 扩展更多执行 Agent 或平台执行能力，但保持 Cybernomads Agent 继续掌握拆分质量门禁。
- 探索任务方案版本对比，让用户清楚看到重拆前后的变化。

### 5.3 对齐项目总方向

本期规划直接对齐 Cybernomads 项目概念介绍中的以下方向：

- “把增长策略变成 AI 可以理解、拆解和执行的任务链路” -> 对应本期 2.1 Cybernomads Agent 任务拆分中心。
- “任务化拆解与自主推进” -> 对应本期 2.2 Agent Review 质量门禁和 2.4 OpenClaw 执行者边界重塑。
- “可配置、可自动执行、可持续迭代的增长流程” -> 对应本期 2.3 任务拆分报告与用户反馈入口。
- “执行可观察与结果可验证” -> 对应本期第 4 节成功标准。
