# CyberNomads 前端开发规范

## 1. 核心原则 (Core Principles)

CyberNomads 前端开发以“可维护、可观测、可演进”为第一原则。当前阶段虽然是本地运行的网页端 MVP，但代码组织不能退化成页面脚本堆叠，必须提前把页面层、交互层、实体层和共享层的边界写清楚。所有前端代码都应优先服务于以下目标：

- 保持页面组装、业务交互、实体表达、基础设施能力之间的单向依赖关系。
- 优先通过目录边界和模块职责控制复杂度，而不是靠约定俗成的人肉记忆。
- 保证本地启动简单，但内部结构能够支撑后续页面扩展、设计系统沉淀和协议调整。
- 保证关键操作有明确反馈，关键状态可观察，关键异常可定位。
- 视觉上继承 [DESIGN.md](./DESIGN.md) 的整体氛围，但不把设计文档当作逐像素复刻手册。

### 1.1 开发技术栈 (Tech Stack)

- **前端框架**：Vue 3
- **开发语言**：TypeScript
- **构建工具**：Vite
- **路由方案**：Vue Router
- **状态管理**：Pinia，遵循“本地状态优先，全局状态按需提升”
- **样式方案**：CSS Variables + SCSS，以设计 token 作为统一样式入口
- **组件策略**：业务组件自研优先；仅允许引入不主导整体视觉风格的轻量依赖或 headless 能力
- **数据通信**：HTTP + JSON
- **长任务状态更新**：默认使用轮询；WebSocket 或 SSE 仅在明确证明必要时引入
- **浏览器本地存储**：仅用于轻量缓存、草稿和界面偏好，不作为业务真实数据源
- **代码质量工具**：ESLint + Prettier + TypeScript 类型检查
- **测试框架**：推荐 Vitest + Vue Test Utils；E2E 推荐 Playwright，按实际落地统一

### 1.2 项目结构与目录规范 (Project Structure)

```text

├── public/                    /**/ 静态资源目录。仅允许放不参与打包的公共文件，如 favicon、robots.txt。禁止放业务配置、脚本或需要版本化管理的设计 token。
├── src/                       /**/ 前端源码根目录。除 `public/` 外的业务代码必须收敛在此目录内。
│   ├── app/                   /**/ 应用启动层。仅允许放应用入口、路由装配、全局 Provider、全局样式接入。可 import `pages/`、`widgets/`、`features/`、`entities/`、`shared/`；禁止被业务层反向 import。
│   │   ├── router/            /**/ 路由表、导航守卫、路由级懒加载配置。禁止写业务请求和页面内部状态逻辑。
│   │   ├── providers/         /**/ Pinia、全局错误处理、运行时上下文注册。禁止沉淀具体业务逻辑。
│   │   └── styles/            /**/ 全局样式入口、reset、字体声明、全局 token 映射。禁止放页面级样式文件。
│   │
│   ├── pages/                 /**/ 路由页面层。一级目录按业务模块命名，二级目录按页面角色拆分。可 import `widgets/`、`features/`、`entities/`、`shared/`；禁止被 `widgets/` 及以下层级 import。
│   │   ├── workspaces/        /**/ 工作区模块页面。承载列表、创建、执行台、任务干预等路由页面。
│   │   │   ├── list/ui/       /**/ 工作区列表页。
│   │   │   ├── create/ui/     /**/ 创建工作区流程页。
│   │   │   ├── detail/ui/     /**/ 工作区执行台页。
│   │   │   └── intervention/ui/**/ 任务干预页。
│   │   ├── assets/            /**/ 资产模块页面。承载列表页与资产编辑器。
│   │   ├── strategies/        /**/ 策略模块页面。承载列表页与策略编辑器。
│   │   ├── accounts/          /**/ 账号模块页面。承载账号池与账号详情认证页。
│   │   └── agents/            /**/ Agent 模块页面。承载总览、空状态与 OpenClaw 配置页。
│   │
│   ├── widgets/               /**/ 页面区块层。承接多个 feature/entity 的组合展示。可 import `features/`、`entities/`、`shared/`；禁止 import `pages/`。
│   │   └── <widget-name>/     /**/ 单个页面区块目录，如日志面板、绑定摘要区。一级按区块命名，二级按类型分。
│   │       ├── ui/            /**/ 区块组件。仅包含区块级展示和交互装配。
│   │       └── model/         /**/ 区块级派生状态和视图逻辑。禁止直接承担跨领域流程编排。
│   │
│   ├── features/              /**/ 业务交互层。封装一次明确的用户动作闭环。可 import `entities/`、`shared/`；禁止 import `pages/`、`widgets/`，也禁止直接依赖其他 feature 的内部实现。
│   │   └── <feature-name>/    /**/ 单个交互特性目录，如连接测试、表单保存、任务启动。一级按特性命名，二级按类型分。
│   │       ├── ui/            /**/ 特性交互组件。
│   │       ├── model/         /**/ 特性状态、校验、组合式函数、Store 切片。
│   │       └── api/           /**/ 特性专用请求封装。仅允许基于 `shared/api` 的基础客户端扩展。
│   │
│   ├── entities/              /**/ 领域实体层。沉淀稳定的业务对象表达、类型和最小展示单元。可 import `shared/`；禁止 import `features/` 及以上层级。
│   │   └── <entity-name>/     /**/ 单个实体目录，如 `account`、`workspace`、`strategy`。一级按实体命名，二级按类型分。
│   │       ├── ui/            /**/ 实体展示组件。禁止承载跨页面流程控制。
│   │       ├── model/         /**/ 类型定义、展示态转换、轻量映射器。
│   │       └── api/           /**/ 实体查询与读写请求。禁止在此处编排页面流程。
│   │
│   └── shared/                /**/ 跨业务共享层。仅存放稳定、无业务归属或弱业务归属的基础能力。禁止 import 上层任何目录。
│       ├── api/               /**/ 请求客户端、拦截器、错误映射、轮询基础设施。禁止放页面专属接口逻辑。
│       ├── config/            /**/ 环境变量读取、运行时配置解析和默认值管理。
│       ├── lib/               /**/ 纯函数工具、格式化工具、日期处理等无副作用工具。
│       ├── types/             /**/ 通用类型和基础协议类型。禁止堆放业务 DTO 大全集。
│       ├── ui/                /**/ 基础 UI 原子组件与无业务归属的复合组件。
│       ├── styles/            /**/ 设计 token、CSS Variables、mixins、spacing 和 typography 规范。
│       ├── hooks/             /**/ 通用组合式函数，仅允许无业务耦合逻辑。
│       ├── mocks/             /**/ 统一 mock 运行时、场景切换和假数据装配入口。禁止页面直接维护本地假数据。
│       └── assets/            /**/ 参与打包的图标、字体、插画等资源。
│
├── tests/                     /**/ 测试根目录。按测试类型分层，禁止放业务源码和一次性调试脚本。
│   ├── unit/                  /**/ 单元测试。聚焦纯函数、Store、组合式函数、基础组件逻辑。
│   ├── integration/           /**/ 集成测试。验证请求层、状态层、组件协作和关键交互闭环。
│   └── e2e/                   /**/ 端到端测试。验证关键用户路径和核心页面联动。
│
├── scripts/                   /**/ 构建、代码生成、校验辅助脚本。禁止放业务逻辑。
└── docs/                      /**/ 前端工程内部补充说明，如组件约定、token 说明。禁止替代 `docs/spec/` 中的正式规范文档。
```

前端源码必须遵守以下单向依赖关系：

- `app -> pages -> widgets -> features -> entities -> shared`
- `shared` 不得反向依赖任何上层目录
- `pages` 只能做组装，不得被下层目录 import
- `features` 之间禁止通过深层相互 import 内部实现；如需共享能力，应下沉到 `entities` 或 `shared`
- 所有跨模块依赖优先通过模块公开入口暴露，禁止使用深层相对路径直接穿透内部文件

### 1.3 Git 与代码流 (Git Workflow)

- **分支模型**：采用 GitHub Flow，所有开发工作从 `main` 拉出短生命周期分支完成
- **分支命名**：统一使用 `feature/*`、`fix/*`、`refactor/*`、`docs/*`
- **提交规范**：采用 Conventional Commits，建议使用 `ui`、`frontend`、`exec`、`workspace` 等明确 scope
- **PR 要求**：涉及界面变更的提交必须附带截图或录屏；涉及轮询、状态管理、表单校验变更的提交必须说明验证方式
- **合并要求**：未通过类型检查、Lint、测试或核心人工验收的改动禁止合并
- **提交原则**：一个提交只表达一个明确意图，禁止把大规模格式化、重构和功能修改混在同一个提交中

### 1.4 命名与代码风格 (Naming & Code Style)

- **目录命名**：目录统一使用 `kebab-case`，一级目录按业务模块或职责命名，禁止按技术角色平铺所有文件
- **Vue 组件命名**：组件文件使用 `PascalCase.vue`；基础组件名称必须表达语义，不允许出现 `TestBox`、`CommonPanel2` 之类占位命名
- **组合式函数命名**：统一使用 `useXxx.ts`；名称必须表达动作或能力，例如 `useTaskPolling`
- **Store 命名**：统一使用 `useXxxStore.ts` 或 `xxx.store.ts`，仓库内保持唯一风格
- **TypeScript 模块导出**：普通 TS 模块优先使用具名导出；禁止默认导出一组含义不明的工具函数
- **路由命名**：路由 path 使用小写中划线风格；路由 name 使用稳定、可读的语义命名
- **样式变量命名**：CSS Variables 使用统一前缀，建议采用 `--cn-*` 语义命名，禁止直接暴露临时色值变量
- **样式书写**：组件局部样式优先使用 token 和语义变量；禁止散落魔法数字、临时色值和无法解释的层级值
- **Lint 工具**：统一使用 ESLint + Prettier；禁止提交未通过 ESLint 的代码
- **类型要求**：禁止提交 TypeScript 类型错误；禁止无意义的 `any` 扩散
- **导入约束**：优先使用别名导入；禁止跨层使用深层相对路径穿透其他模块内部实现
- **注释原则**：只解释边界、取舍和不直观原因；禁止写“代码翻译式注释”

### 1.5 其他开发事项 (Other Conventions)

- **设计系统基线**：视觉实现以 [DESIGN.md](./DESIGN.md) 为母方向。必须保持整体氛围、层次和材质一致，但允许在特殊模块中做局部创新，不要求逐像素复刻。
- **页面层级约束**：一级导航仅允许存在 `工作区`、`资产`、`策略`、`账号`、`Agent` 五个主模块。工作区执行台、任务干预页、Agent 空状态页和 OpenClaw 配置页均属于模块内部子页面。
- **Mock 数据约束**：必须通过 `shared/mocks` 和模块 API 边界组织统一 mock 数据，禁止页面直接维护分散 JSON 假数据。
- **UI 组件库约束**：禁止直接以重型通用后台组件库作为页面主骨架。确需引入第三方能力时，优先选择 headless 或弱样式方案。
- **页面职责约束**：页面组件只负责布局、装配和路由级编排，禁止直接散落接口细节和复杂业务状态转换。
- **状态管理约束**：遵循“组件内优先、页面级次之、全局 Store 最后”的原则，禁止把所有状态无差别塞进 Pinia。
- **请求封装约束**：所有接口请求必须经过 `shared/api` 或模块 `api/` 封装，禁止在模板文件和页面 `setup` 中直接手写重复请求逻辑。
- **轮询约束**：长任务状态和日志更新默认使用轮询；必须具备启动、去重、停止和页面卸载清理机制。未完成论证前禁止上 WebSocket 或 SSE。
- **浏览器存储约束**：`localStorage`、`sessionStorage` 仅可用于草稿、筛选条件、UI 偏好和轻量缓存，不得作为业务真实数据源。
- **Markdown 约束**：Markdown 编辑以稳定编辑、基础预览和安全渲染为目标；禁止在 MVP 阶段引入复杂富文本平台化方案。
- **桌面优先约束**：当前前端以桌面宽屏网页体验为主，不以移动端完整适配为交付前提；但不能出现桌面端基本可用性问题。
- **异常反馈约束**：保存、创建、测试连接、启动任务、加载列表等关键动作必须有清晰的成功、失败和处理中反馈。

## 2. 治理与维护 (Governance & Maintenance)

### 2.1 测试规范 (Testing Strategy)

- **测试重点**：优先覆盖页面组装层以外的高风险区域，包括状态管理、表单校验、请求封装、轮询逻辑和关键组件协作
- **单元测试要求**：纯函数、组合式函数、Store、复杂展示态转换必须具备单元测试
- **组件测试要求**：基础组件和关键交互组件应覆盖核心状态切换、事件触发和异常反馈
- **集成测试要求**：请求层、状态层和组件装配之间的关键协作路径必须具备集成测试
- **E2E 要求**：至少覆盖一条关键用户主路径和一条关键失败路径
- **回归要求**：修复轮询泄漏、状态错乱、重复提交、渲染安全、表单校验等问题时，必须补对应回归测试
- **覆盖率原则**：不强制追求统一百分比指标，但所有核心变更都必须给出自动化测试或明确人工验收记录
- **最小验收标准**：关键模块至少覆盖“正常流程 + 一个失败分支 + 一个边界条件”

### 2.2 部署与 CI/CD (Deployment & CI/CD)

- **环境划分**：当前至少区分本地开发环境与持续集成校验环境；如后续出现可分发形态，再补充 release 环境定义
- **本地启动要求**：前端必须支持命令启动，并能稳定连接本地后端服务或本地 mock 环境
- **配置管理**：所有环境变量必须通过统一配置入口读取；禁止把接口地址、开关和运行参数硬编码进组件内部
- **构建要求**：每次合并前至少通过类型检查、Lint、测试和构建产物验证
- **CI 最低流程**：安装依赖 -> 类型检查 -> Lint -> 单元测试 / 集成测试 -> 构建
- **发布约束**：未通过质量门槛的前端版本禁止进入可分发状态
- **产物约束**：构建产物必须能被本地后端或统一静态资源入口稳定承载，禁止依赖隐式本地路径

### 2.3 监控与告警 (Monitoring & Alerting)

- **日志规范**：前端错误日志必须包含足够上下文，至少包括模块来源、动作语义和失败摘要；如后端返回请求标识，应保留关联能力
- **错误分级**：区分用户可恢复错误、接口错误、渲染错误和系统异常，避免所有错误都以相同方式呈现
- **本地诊断优先**：当前阶段以本地可排查性为主，开发环境中的错误输出必须能帮助快速定位问题
- **噪音控制**：禁止在生产构建中保留无意义 `console.log`、调试痕迹或重复报错刷屏逻辑
- **任务观察约束**：长任务页面出现轮询失败、请求超时或状态不一致时，必须给出用户可理解的降级提示和重试入口
- **后续扩展预留**：如未来接入远端部署或统一监控，前端错误结构应能平滑映射到集中式日志与告警系统
