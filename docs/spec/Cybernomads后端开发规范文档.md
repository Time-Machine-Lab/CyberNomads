# Cybernomads后端开发规范文档

## 1. 核心原则 (Core Principles)

Cybernomads 后端开发以“可演进、可替换、可读懂”为第一原则。当前阶段虽然是本地运行的单体后端，但代码组织必须提前为后续多平台、多 Agent、多存储介质演进留出边界。所有后端代码都应优先服务于以下目标：

- 保持业务层、Agent 层、平台能力层、存储层之间的低耦合。
- 优先使用模块边界和接口抽象控制复杂度，而不是靠临时脚本堆叠功能。
- 保证本地开发启动简单，但内部结构不能因为 MVP 而失去扩展能力。
- 所有新增代码都应能被后续维护者快速理解、测试和替换。

### 1.1 开发技术栈 (Tech Stack)

- **后端语言**：TypeScript
- **运行时**：Node.js LTS
- **服务形态**：本地启动的 HTTP 服务，负责承载 API、调度、日志查询和前端访问入口
- **前端协作方式**：前后端分目录开发，后端提供本地访问入口与接口契约
- **结构化存储**：当前默认 SQLite，后续允许替换为 MySQL、Redis 或其他介质
- **非结构化存储**：当前默认本地文件系统，后续允许替换为对象存储或远端文件服务
- **通信协议**：HTTP + JSON
- **测试框架**：推荐 Vitest 或 Jest，按仓库最终选型统一
- **代码质量工具**：ESLint + Prettier

### 1.2 Git 与代码流 (Git Workflow)

- **分支模型**：采用 GitHub Flow，所有开发工作从 `main` 拉出短生命周期分支完成
- **分支命名**：统一使用 `feature/*`、`fix/*`、`refactor/*`、`docs/*`
- **提交规范**：采用 Conventional Commits，例如 `feat: xxx`、`fix: xxx`、`refactor: xxx`、`docs: xxx`
- **合并要求**：涉及业务编排、Agent 适配、存储抽象变更的提交必须经过 Code Review
- **提交原则**：一个提交只表达一个明确意图，避免把重构、修复和格式化混在同一个提交中

### 1.3 命名与代码风格 (Naming & Code Style)

- **目录命名**：目录统一使用小写中划线或小写单词，避免中英文混用和随意缩写
- **文件命名**：TypeScript 文件统一使用 `kebab-case`；类型、接口、类名使用 `PascalCase`
- **函数命名**：函数使用 `camelCase`，名称必须体现动作语义，避免使用模糊名称如 `handleData`、`processThing`
- **接口命名**：端口接口统一使用具备语义的 `*Port` 或 `*Repository` 命名
- **适配器命名**：适配器统一使用 `*Adapter` 命名，例如 `OpenClawAdapter`
- **API 路径**：统一使用小写中划线风格，例如 `/api/traffic-works`
- **Lint 规则**：禁止提交未通过 ESLint 的代码
- **格式规则**：统一使用 Prettier 自动格式化，禁止手工制造风格差异
- **导入约束**：禁止跨层级随意反向依赖，例如适配器层不得反向依赖业务域层的内部实现细节

### 1.4 其他开发事项 (Other Conventions)

- **架构边界优先**：业务代码必须优先依赖抽象接口，不得直接把 SQLite、文件系统、OpenClaw 或具体平台脚本写进业务核心流程
- **模块化单体原则**：当前虽然是单体应用，但代码必须按模块边界拆分，而不是把所有逻辑堆进一个服务文件
- **平台能力约束**：平台支持应通过平台能力抽象承接，当前即使只实现 B 站，也不得把 B 站能力直接写死到所有业务流程中
- **Agent 调用约束**：subagent 不允许直接修改数据库，只能通过系统工具或后端暴露的受控接口回写状态和日志
- **配置管理**：所有环境配置必须通过配置文件或环境变量注入，禁止把端口、路径、连接信息硬编码进业务代码
- **日志分层**：系统日志、调度日志、任务执行日志和错误日志应分层组织，避免把所有输出混在同一类日志里
- **异常处理**：所有可预期错误必须转成可识别的业务错误或基础设施错误，禁止直接向上抛出无上下文异常
- **注释原则**：只在边界复杂、抽象不明显或存在重要取舍时写注释，避免写“代码翻译式注释”
- **文档同步**：只要涉及模块边界、端口定义或开发约束的变化，必须同步更新相应规范文档

## 2. 治理与维护 (Governance & Maintenance)

### 2.1 测试规范 (Testing Strategy)

- **测试重点**：优先覆盖业务编排层、任务调度层、Agent 适配边界和存储抽象边界
- **单元测试要求**：核心业务服务和任务调度逻辑必须具备单元测试
- **集成测试要求**：涉及端口与适配器协作的关键路径必须具备集成测试
- **Mock 原则**：单元测试中允许 Mock AgentPort、MetadataStoragePort、ArtifactStoragePort；集成测试中应尽量验证真实协作路径
- **回归要求**：修复任务状态流转、日志写入、对象绑定解析相关问题时，必须补回归测试
- **最小验收标准**：所有核心模块至少覆盖“正常流程 + 一个失败分支 + 一个边界条件”

### 2.2 部署与 CI/CD (Deployment & CI/CD)

- **环境划分**：当前至少区分本地开发环境与持续集成校验环境；后续如出现远端发布形态，再补充 release 环境定义
- **本地启动方式**：后端必须支持命令启动，并以固定入口启动本地 HTTP 服务，供浏览器访问
- **构建要求**：每次合并前至少保证通过类型检查、Lint 和测试
- **CI 最低流程**：安装依赖 -> 类型检查 -> Lint -> 单元测试 / 集成测试 -> 构建产物验证
- **发布约束**：未通过类型检查或测试的版本禁止进入可分发状态
- **兼容性原则**：所有后端启动脚本必须尽量保持对团队主流开发环境的兼容，避免依赖隐式本地工具链

### 2.3 监控与告警 (Monitoring & Alerting)

- **日志规范**：每条错误日志必须附带足够上下文，至少包含模块来源、动作语义和失败摘要
- **任务追踪**：与引流工作或任务执行相关的日志，必须具备可关联的工作或任务上下文标识
- **错误分级**：区分业务错误、适配器错误、配置错误和系统错误，避免所有错误都以同一严重度输出
- **本地诊断优先**：当前阶段以本地可排查性为主，日志输出应优先服务开发者定位问题
- **后续扩展预留**：如未来接入远端运行形态，日志结构应能平滑接入集中式日志与告警系统
