# Cybernomads后端开发规范文档

## 1. 核心原则 (Core Principles)

Cybernomads 后端在 MVP 阶段采用“先跑通主链路、再逐步长大”的开发原则。当前项目是单机本地、单用户运行的模块化单体，因此规范的目标不是把目录预拆到很细，而是用最少的层级把边界留清楚，让团队能快速开发，同时不阻碍后续切换 Agent、平台能力和存储介质。

- 业务代码优先依赖抽象边界，不直接耦合 OpenClaw、SQLite、本地文件系统或具体平台脚本。
- 目录结构服务于当前开发效率，禁止为了“未来可能会复杂”而过度拆层。
- 先保证引流工作、任务调度、日志回写等核心链路清晰，再补充细化分层。
- 当前只实现 B 站交互能力，但代码组织必须保留多平台扩展口。
- 当前只接 OpenClaw，但 Agent 接入方式必须可替换。
- 存储层必须解耦，后续替换为 MySQL、Redis 或对象存储时，不应倒逼重写业务层。

### 1.1 开发技术栈 (Tech Stack)

- **前端框架**：Vue
- **后端框架**：TypeScript + Node.js
- **运行形态**：本地命令启动后端服务，开放 HTTP 端口供前端访问
- **数据存储**：SQLite + 本地文件系统
- **通信协议**：HTTP + JSON
- **Agent 接入**：当前接入 OpenClaw，后续保留远端 Agent / 本地 Agent 的替换能力
- **测试框架**：Vitest
- **代码质量工具**：ESLint + Prettier

### 1.2 项目结构与目录规范 (Project Structure)

当前目录应保持“边界清楚，但层级克制”。MVP 阶段不强制每个模块内部立即拆成 `application/domain/http`，而是先允许模块内用轻量文件组织；只有当某个模块真实变复杂后，才继续向下细分。

```text
cybernomads-backend/
├── src/                           /**/ 后端源码根目录，只放生产代码；禁止放测试夹具、运行产物和临时脚本。
│   ├── index.ts                   /**/ 进程入口文件，只负责启动应用、装配依赖和暴露服务入口，不写具体业务逻辑。
│   ├── app/                       /**/ 应用级编排目录，放启动装配、调度循环、任务分发等跨模块流程；不放平台脚本和存储实现。
│   ├── modules/                   /**/ 业务模块目录，按业务概念拆分代码；模块内部默认轻量组织，避免过早分层。
│   │   ├── agent-access/          /**/ Agent 接入模块，放 Agent 配置、连接、Skill 安装流程等相关代码。
│   │   ├── observation/           /**/ 观察与日志模块，放执行日志、过程记录、结果查看相关代码。
│   │   ├── products/              /**/ 产品模块，放产品资料、Markdown 内容和产品提示词管理代码。
│   │   ├── resource-binding/      /**/ 对象绑定模块，放策略对象引用、资源映射和绑定解析相关代码。
│   │   ├── strategies/            /**/ 策略模块，放策略包组合、策略编译和策略内容管理代码。
│   │   ├── tasks/                 /**/ 任务模块，放任务定义、任务状态和任务执行条件相关代码。
│   │   └── traffic-works/         /**/ 引流工作模块，放引流工作创建、启动、暂停、更新和任务生成入口代码。
│   ├── ports/                     /**/ 系统抽象边界目录，只放接口、契约和协议类型；禁止写任何具体实现。
│   ├── adapters/                  /**/ 外部实现目录，放 Agent、平台、Skill、存储等适配器实现；禁止承载业务规则。
│   │   ├── agent/                 /**/ Agent 适配实现目录，例如 OpenClaw；只负责调用协议适配，不负责业务决策。
│   │   ├── platform/              /**/ 平台能力实现目录，当前先实现 B 站交互脚本，同时保留多平台扩展方式。
│   │   ├── skill/                 /**/ Skill 安装与管理实现目录，负责向 Agent 安装或同步 Cybernomads 所需 Skill。
│   │   └── storage/               /**/ 存储实现目录，负责 SQLite 与文件系统读写，不承载业务判断。
│   └── shared/                    /**/ 稳定通用代码目录，放错误定义、日志封装、工具函数、共享类型；禁止塞入具体业务逻辑。
├── tests/                         /**/ 测试目录，只放测试代码与测试数据；禁止放生产源码。
│   ├── unit/                      /**/ 单元测试目录，重点覆盖调度逻辑、任务状态流转和模块级服务逻辑。
│   ├── integration/               /**/ 集成测试目录，重点覆盖 ports 与 adapters 协作、服务启动链路和关键主流程。
│   └── fixtures/                  /**/ 测试夹具目录，放测试输入、模拟日志、示例 Markdown 和样例数据。
├── scripts/                       /**/ 辅助脚本目录，放开发、迁移、诊断、初始化脚本；禁止写正式业务逻辑。
├── storage/                       /**/ 本地运行时数据目录，只放 SQLite 文件、日志和执行产物；禁止放源码。
│   ├── sqlite/                    /**/ SQLite 数据文件目录，后续如更换存储介质，可整体替换该实现。
│   └── artifacts/                 /**/ 运行产物目录，放日志、Markdown 产物、执行输出等文件。
├── package.json
├── tsconfig.json
├── tsconfig.build.json
├── vitest.config.ts
├── eslint.config.mjs
└── .env.example
```

目录补充约束如下：

- `src/modules/*` 内部在 MVP 阶段优先使用 `service.ts`、`model.ts`、`controller.ts`、`types.ts` 这类轻量文件组织。
- 只有当某个模块出现明显复杂度，例如规则密集、输入输出协议很多、领域状态机复杂时，才允许继续细分子目录。
- `src/ports` 中定义的是系统长期稳定边界，命名统一使用 `*Port`，例如 `AgentPort`、`MetadataStoragePort`。
- `src/adapters` 中的代码必须是“替换实现”，不允许反向定义业务主流程。
- `src/shared` 只放跨模块复用且相对稳定的代码；如果一段代码只服务一个业务模块，应放回模块内部。


### 1.4 命名与代码风格 (Naming & Code Style)

- **目录命名**：统一使用小写英文或 `kebab-case`
- **文件命名**：统一使用 `kebab-case.ts`
- **类型命名**：接口、类型、类名统一使用 `PascalCase`
- **函数命名**：统一使用 `camelCase`，函数名要体现动作语义
- **接口命名**：抽象边界统一使用 `*Port`，适配实现统一使用 `*Adapter`
- **API 路径命名**：统一使用小写中划线，例如 `/api/traffic-works`
- **格式规范**：提交前必须通过 ESLint 和 Prettier
- **导入约束**：禁止跨层反向依赖，`modules` 不得直接依赖具体 `adapters` 实现

### 1.5 其他开发事项 (Other Conventions)

- 存储层必须通过抽象接口访问，禁止把 SQLite SQL 或文件路径逻辑直接写进业务流程。
- subagent 不允许直接修改数据库，只能通过受控工具或后端接口回写任务状态和日志。
- 当前任务调度采用线程轮询决策，相关代码统一放在 `src/app` 编排层，不要散落到各业务模块里。
- 策略、策略包、产品内容本质上都是 Prompt 资产，相关持久化与组装逻辑应放在业务模块中，不应直接下沉到适配器。
- 当前只做 B 站交互脚本实现，但平台能力接口的命名和输入输出不应写死 B 站语义。
- 如未来某个模块复杂度明显上升，可以在该模块内部再引入更细的子层，而不是提前在所有模块统一预拆。
