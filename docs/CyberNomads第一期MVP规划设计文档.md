# CyberNomads第一期MVP规划设计

## 1. 本期目标与定位

### 1.1 本期愿景陈述
本期（MVP 阶段）结束后，用户可以在本地启动 CyberNomads，通过网页完成 Agent 接入、B 站账号登录、产品与策略配置，并在工作区中发起一次可观测的 AI 引流任务。

### 1.2 与项目总方向的对齐
本期直接对齐 [Cybernomads项目概念介绍文档](D:/Code/Project/Github/CyberNomads/docs/Cybernomads项目概念介绍文档.md) 中第 4.1 节“第一期核心交付物”定义的本地化完整产品形态，重点落地账号接入、产品与策略管理、工作区绑定和 AI 自主执行能力。同时，本期也对齐第 3.2 节“典型使用场景”中的新品冷启动引流、低预算日常账号运营和高意向用户私信跟进三类核心场景，为后续更高阶能力提供首个可运行闭环。

### 1.3 本期核心要解决的问题
- 问题：用户即使有明确的增长思路，也缺少一个统一入口把 Agent、平台账号、产品内容和策略组织成可执行的工作区。
  期望：用户能在一个本地产品中完成接入、配置和启动，不需要在多个工具之间切换。
- 问题：用户当前很难把“引流策略”真正变成可执行、可观测的 AI 任务流程。
  期望：用户启动任务后，能够看到任务流状态、执行进展和统一日志，从而确认策略已经被真正跑起来。

## 2. 功能模块规划

### 2.1 启动与 Agent 接入中心

**模块一句话价值**：让用户先把 CyberNomads 跑起来，并完成首个可用 Agent 的接入。

#### 2.1.1 背景与目的
MVP 阶段的第一道门槛不是增长策略本身，而是让用户能顺利启动产品，并把一个可用的 Agent 接入到系统中，作为后续所有任务执行的基础入口。

#### 2.1.2 核心功能描述
1. 用户启动本地服务后，通过浏览器进入 CyberNomads 首页。
2. 用户从导航进入 Agent 接入页，查看当前可选的接入方式。
3. 用户选择 OpenClaw 作为当前可用接入项并填写连接信息。
4. 用户发起连接测试，并看到当前 Agent 是否可用。
5. 用户在接入完成后看到系统将为该 Agent 准备所需 Skill 的提示。
6. 用户完成接入后返回工作区列表，继续后续配置流程。

#### 2.1.3 预期交付形态
该模块以独立设置页呈现。页面上方显示当前 Agent 接入状态，中间为接入方式选择和连接信息填写区域，下方展示连接结果与 Skill 准备提示。当前页面只开放 OpenClaw 选项，但界面会保留后续新增其他 Agent 接入方式的位置。

#### 2.1.4 本期不做（模块级边界）
- 不提供多个 Agent 并行接入
- 不提供复杂鉴权与权限管理
- 不提供 Skill 市场或自定义安装流
- 不提供 Agent 自动切换与容灾

### 2.2 账号、产品与策略配置中心

**模块一句话价值**：让用户把“谁来执行、为谁引流、按什么策略做”配置完整。

#### 2.2.1 背景与目的
在任务真正运行之前，用户必须先准备好平台账号、产品内容和策略内容。MVP 要解决的是这三类增长资产的统一配置问题，而不是一开始就做复杂的运营后台。

#### 2.2.2 核心功能描述
1. 用户进入账号页，新增并登录一个 B 站账号。
2. 用户在账号列表中查看已登录账号并确认其可用于后续工作区。
3. 用户创建一个产品，并用 Markdown 编写产品介绍与宣传重点。
4. 用户创建一条策略，并用 Markdown 编写给 AI 使用的策略内容。
5. 用户可以基于预设策略模板快速创建策略。
6. 用户也可以直接新建自定义策略内容并保存。

#### 2.2.3 预期交付形态
该模块以多标签配置页呈现，主要分为“账号”“产品”“策略”三个入口。账号页以列表方式管理多个账号；产品与策略页采用编辑器形态，用户可以直接编写 Markdown 内容，并在保存后返回各自列表页继续管理。

#### 2.2.4 本期不做（模块级边界）
- 不支持 B 站以外的平台账号登录
- 不提供可视化策略拖拽编排
- 不提供品牌人格与语气系统
- 不提供策略 A/B 实验能力

### 2.3 工作区与任务执行台

**模块一句话价值**：让用户真正发起一次策略驱动的 AI 引流执行，并清楚看到任务是如何运行的。

#### 2.3.1 背景与目的
如果产品只能完成配置而无法展示执行过程，它就只是一个后台。MVP 必须把产品、策略和账号真正绑定成工作区，并让用户看到任务流、状态变化和执行日志，证明策略已经转化为可运行的 AI 行动链路。

#### 2.3.2 核心功能描述
1. 用户在工作区中绑定 1 个产品、1 条策略和 1 个或多个账号。
2. 用户进入工作区详情页，查看当前绑定关系和执行入口。
3. 用户点击启动，发起一次 AI 引流任务执行。
4. 用户在执行台查看任务流可视化和状态面板。
5. 用户在日志区查看搜索、评论、私信等执行记录。
6. 用户在任务完成后查看整体状态与最新产出。

#### 2.3.3 预期交付形态
工作区列表将作为产品首页。用户进入某个工作区后，会看到上方的产品、策略和账号绑定摘要，中部为任务流状态面板，底部为按时间排序的执行日志区域。日志统一承接 AI 在执行中产生的评论、私信、搜索和阶段性结果，不在本期拆成复杂的数据中心。

#### 2.3.4 本期不做（模块级边界）
- 不支持任务暂停或手动停止
- 不提供复杂任务编排与编辑器
- 不提供 CRM 式线索管理能力
- 不提供复杂归因与复盘分析

## 3. 本期整体交付形态

MVP 阶段结束后，用户通过命令启动 CyberNomads 后，会在浏览器中进入一个以“工作区列表”为首页的本地产品。如果用户尚未完成接入，首页会以空状态提示其先进入 Agent 接入页和账号配置页。用户完成 OpenClaw 接入后，可以在账号页登录 B 站账号，在产品页创建产品并填写 Markdown 介绍，在策略页创建策略并填写给 AI 使用的 Markdown 提示内容。

当这些基础资产准备完成后，用户返回工作区列表，新建一个工作区，并在其中绑定 1 个产品、1 条策略和 1 个或多个账号。进入工作区详情后，页面顶部展示绑定摘要和启动入口，中间展示任务流可视化与状态面板，底部展示统一日志。用户由此可以从“启动产品”一路走到“看到 AI 真正开始执行任务”，并在同一个产品中持续观察执行进展，而无需跳转到外部工具。

## 4. 本期成功标准

| 验收维度 | 成功条件 | 验证方式 |
| :--- | :--- | :--- |
| 本地启动可用性 | 用户启动服务后，可通过浏览器正常访问 CyberNomads 首页 | 产品走查 |
| Agent 接入完整性 | 用户可在接入页选择 OpenClaw、填写连接信息并看到接入成功状态 | QA 执行接入流程 |
| Agent 能力准备 | 用户完成接入后，可明确感知系统已为当前 Agent 准备运行本产品所需的 Skill | 产品走查 |
| 账号配置完整性 | 用户可完成 1 个 B 站账号登录，并在账号列表中看到该账号 | QA 执行账号登录流程 |
| 产品与策略配置 | 用户可分别创建 1 个产品和 1 条策略，并保存 Markdown 内容 | QA 执行创建流程 |
| 工作区绑定能力 | 用户可创建工作区，并绑定 1 个产品、1 条策略和至少 1 个账号 | 产品走查 |
| 执行台可观测性 | 用户启动任务后，可看到任务流状态面板和统一日志区域 | QA 执行一次完整任务 |
| 日志结果可验证性 | 日志中至少可观测到搜索、评论、私信等任务记录中的一种或多种 | 产品走查与样例验证 |
| 用户路径完整性 | 一个首次接触产品的内部成员，可在 10 分钟内完成“接入 -> 配置 -> 建立工作区 -> 启动任务”全流程 | 内部试用反馈 |

## 5. 参考约束与衔接说明

### 5.1 上期回顾（选填）
本期为首期启动，无上期。

### 5.2 下期预告（选填）
下期规划方向可围绕以下能力逐步展开：
- 扩展更多平台接入方式，而非只支持 B 站
- 扩展更多 Agent 接入方式，而非只支持 OpenClaw
- 引入归因与复盘能力，解释为什么这样执行、结果如何、下一步如何调整
- 引入线索雷达、品牌人格系统与策略实验室等更高阶能力

### 5.3 对齐项目总方向
本期规划直接对齐 [Cybernomads项目概念介绍文档](D:/Code/Project/Github/CyberNomads/docs/Cybernomads项目概念介绍文档.md) 中的以下章节：
- 第 4.1 节“第一期核心交付物” -> 对应本期第 2.1 节启动与 Agent 接入中心、第 2.2 节账号、产品与策略配置中心、第 2.3 节工作区与任务执行台
- 第 2.2 节“关键解决方案拆解”中的“策略包驱动的增长执行” -> 对应本期第 2.2 节和第 2.3 节
- 第 2.2 节“关键解决方案拆解”中的“任务化拆解与自主推进” -> 对应本期第 2.3 节工作区与任务执行台
- 第 3.2 节“典型使用场景” -> 对应本期整体交付路径中的新品冷启动、日常账号运营和私信跟进的基础支撑能力
- 第 4.3 节“成功指标与商业价值” -> 对应本期第 4 节中的执行可观测性与日志结果验证

需要强调的是，本期文档只定义面向用户可感知的产品方向和模块边界，不将当前的 Agent 接入方式、平台入口和本地数据承载方式视为最终形态。相关能力在本期需保留扩展空间，以支持后续多平台、多 Agent 和不同数据承载方式的演进。
