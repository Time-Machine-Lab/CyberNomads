## Context

CyberNomads 的第一期前端目标不是单张设计稿展示，而是基于现有 UI 资产构建一个能跑通完整产品主路径的网页端。现状存在三个约束：

- 设计稿来源分散，且混合了一级入口页、编辑页、详情页、模块内部子页和空状态页。
- `cybernomads-ui` 当前仅有基础脚手架与单一首页路由，尚未形成模块化页面结构。
- 项目已经定义了前端开发规范和页面整合设计，后续实现必须以这些文档为来源，而不是按截图逐张硬编码。

这次 change 的核心不是新增某个孤立页面，而是建立前端实现的结构性基线：稳定信息架构、稳定主链路、稳定 Mock 运行时。

## Goals / Non-Goals

**Goals:**

- 将分散设计稿归并为五个稳定一级模块及其内部子页面层级。
- 为 `cybernomads-ui` 建立与页面层级一致的路由结构和目录骨架。
- 统一前端领域模型与 Mock 数据入口，保证不同页面消费同一套业务对象表达。
- 明确首启空状态、资源准备、工作区执行台和任务干预等关键页面之间的流转关系。
- 让后续实现任务可以按模块和依赖顺序推进，而不是按设计稿编号推进。

**Non-Goals:**

- 不在本 change 中定义后端接口细节、数据库结构或持久化协议。
- 不要求逐像素复刻设计稿中的所有装饰和动画细节。
- 不引入真实 WebSocket/SSE、复杂富文本系统或新的第三方重型 UI 框架。
- 不在本设计中覆盖移动端完整适配或多租户、多用户场景。

## Decisions

### 1. 以五个一级模块作为唯一稳定导航骨架

决定保留 `工作区`、`资产`、`策略`、`账号`、`Agent` 五个一级模块，其他设计稿全部作为子页、配置页或空状态页挂载到这五个模块内部。

原因：

- 这五个模块已经覆盖 MVP 的真实业务对象。
- `_6`、`_1`、`_9` 这类页面本质上依赖父上下文，强行提升为一级页会让导航与数据流都变形。
- 统一一级导航后，后续权限、面包屑、空状态、返回路径都会更稳定。

替代方案：

- 直接按设计稿逐页建立一级路由。
  - 放弃原因：会把内部页误做成主模块，后期回收成本高。

### 2. 以工作区详情页承载执行台，而不是单独建设执行模块

决定把 `_6` 对应的执行视图放在 `/workspaces/:workspaceId` 或 `/workspaces/:workspaceId/execution` 下，作为工作区详情中的核心视图。

原因：

- 执行台天然依赖工作区上下文、绑定资源与任务集合。
- 工作区是产品主入口，执行是工作区生命周期的一部分，不是独立领域。

替代方案：

- 独立建设 `/execution` 顶级模块。
  - 放弃原因：会割裂工作区与任务上下文，造成额外的路由与状态同步复杂度。

### 3. 以统一领域模型组织 Mock，而不是按页面拆 JSON

决定统一沉淀 `workspace`、`asset`、`strategy`、`account`、`agentNode`、`taskRun`、`interventionRecord` 七类核心模型，并通过 mock adapter 暴露页面所需数据。

原因：

- 页面之间共享对象语义，必须先共享模型，后共享视图。
- 按页面拆 JSON 会导致相同业务对象在不同页面出现不同字段结构，后续真实接口接入成本高。

替代方案：

- 每个页面各自维护本地 mock 数据。
  - 放弃原因：短期快，长期不可维护。

### 4. 先覆盖主链路，再补装饰细节

决定实现顺序先覆盖三类主链路：首次进入产品、基础资源准备、工作区创建与执行观察；设计细节和复杂动画放在后续迭代。

原因：

- 这是 MVP 真正可验收的路径。
- 当前最大风险不是视觉不够满，而是页面之间不能正常串起来。

替代方案：

- 优先复刻单页视觉细节。
  - 放弃原因：容易形成一批孤立页面，无法构成完整产品。

## Risks / Trade-offs

- [Risk] 设计稿之间存在文案和术语不完全统一，可能影响路由命名和模块认知。 → Mitigation：统一以工作区、资产、策略、账号、Agent 为正式术语，其余视为视觉稿中的局部表达。
- [Risk] 先用 Mock 驱动会掩盖部分真实接口约束。 → Mitigation：所有 mock 入口通过统一 API 抽象暴露，后续真实接口仅替换 adapter。
- [Risk] 工作区执行台包含图形流、日志、干预等多类交互，初版复杂度较高。 → Mitigation：先实现页面壳、状态面板和基础日志/干预闭环，再逐步增强图形细节。
- [Risk] 一级模块过少可能让某些页面显得“深”。 → Mitigation：通过 breadcrumb、页面标题和返回路径强化父子关系，而不是增加一级导航数量。

## Migration Plan

1. 先在 OpenSpec 中固定 proposal、design、specs、tasks，作为实现基线。
2. 按变更任务在 `cybernomads-ui` 中建立一级模块路由和页面目录骨架。
3. 建立统一实体类型、mock 数据和 mock API 入口。
4. 按主链路逐步接入页面壳与区块。
5. 每完成一条链路，补齐对应测试与文档核对。

当前没有线上发布或回滚要求；此变更面向本地开发环境与后续实现阶段。

## Open Questions

- 工作区详情是否直接使用 `/workspaces/:workspaceId` 作为执行台默认视图，还是显式保留 `/execution` 子路由作为默认跳转目标。
- 首次进入产品时，首页是否始终定位为工作区列表，还是在无 Agent 时直接跳转到 Agent 模块空状态页。
- 任务干预页最终采用整页路由还是右侧抽屉式子视图；当前 change 仅要求有明确子页面承载，不强制限定最终 UI 形式。
