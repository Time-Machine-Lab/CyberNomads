## Context

CyberNomads 第一轮前端实现已经完成模块化路由、基础页面骨架和统一 mock 运行时，但结果更接近“结构样机”而不是“设计还原版产品”。用户当前要求已经明确：

- 目标是网页端桌面体验，视觉效果要与参考设计稿高度一致，而不是只保留功能闭环。
- 全局壳层需要统一，左侧导航以 `_4` 的结构为主参考，使用中文导航、支持折叠，并统一替换为 `CyberNomads` 品牌表达。
- 关键页面不只是列表和表单，还包括工作区执行台、Agent 空状态和 OpenClaw 配置页，这些页面决定整体视觉语言。
- 当设计资源缺失时，可以先用截图型占位资源或近似图形替代，但布局、层次、氛围和交互位置不能走样。

当前约束也很明确：

- `docs/spec/Cybernomads前端开发规范.md` 要求继续保持分层目录、统一 mock 边界和桌面优先策略。
- `docs/spec/DESIGN.md` 给出了深色控制台、无硬边分割线、霓虹蓝高亮和非对称布局的设计母方向。
- 顶层 `docs/api` 和 `docs/sql` 目前没有可修改的正式契约文件，所以本次变更不能把问题转移为后端接口变更。

## Goals / Non-Goals

**Goals:**

- 为 CyberNomads 定义一套可执行的高保真前端重建方案，覆盖统一壳层、关键页面视觉、交互补全和 mock 视觉支撑。
- 明确哪些参考稿用于全局壳层，哪些用于模块内部页面，避免再次把结构整理和视觉重建混在一起。
- 把现有功能骨架提升到可验收的高保真界面标准，同时保留前一轮已经确认的信息架构和主链路。
- 让后续实现任务可以按视觉基础设施、全局壳层、关键页面、其余模块和验收校准分阶段推进。

**Non-Goals:**

- 不在本次设计中引入新的后端 API、数据库结构或真实在线数据通道。
- 不要求移动端完整适配；本次只要求常见桌面分辨率下的稳定还原。
- 不追求把所有缺失品牌素材在本轮一次性补齐；允许先使用截图占位和可替代图形。
- 不把执行台做成完整可拖拽编排器；优先保证查看、切换、日志和干预闭环可用。

## Decisions

### 1. 使用统一桌面壳层，页面内部再做高保真分化

决定所有主模块复用一套统一的桌面壳层：固定左侧导航、顶部模块栏或上下文栏、内容画布区。左侧导航的结构、尺寸和视觉语义以 `_4` 为主参考，但品牌命名统一为 `CyberNomads`，图标语言参考 `agent_openclaw`。

原因：

- 用户明确要求切换 tab 时整体需要一致，只让内部页面面板差异化。
- 高保真重建最大风险是每个页面各自实现一套壳，最后导致样式碎裂。
- `_4` 的侧栏结构最适合作为全局壳层基线，既有明确的导航组织，也适配折叠态。

替代方案：

- 直接沿用每张设计稿原始壳层。
  - 放弃原因：不同设计稿的导航体系和品牌表达不一致，无法形成真实产品壳层。

### 2. 以关键参考页定义视觉语言，再向其他模块扩散

决定优先把 `_4`、`_6`、`_9`、`_11` 和 `agent_openclaw` 作为视觉锚点。它们分别定义创建流壳层、执行台密度、空状态表达、工作区卡片列表和设置页材质。其他页面在不改变 IA 的前提下，沿用这几类视觉规则做同风格重构。

原因：

- 这些页面覆盖了主要视觉模式：步骤型、卡片型、图形控制台型、空状态型、配置型。
- 用户要求“都要完全做完”，但实际高保真基线需要先从代表性页面抽象，否则后续实现会继续按截图碎片前进。

替代方案：

- 逐页独立定义视觉标准。
  - 放弃原因：重复工作高，且容易在间距、标题层级和面板材质上出现多套标准。

### 3. 用视觉 token + 页面配方，而不是逐页面硬编码

决定把高保真重建拆成两层：

- 共享层沉淀颜色、字号、边距、发光、幽灵描边、面板背景和状态色 token。
- 页面层沉淀“配方”，例如工作区列表卡片、执行台节点卡、空状态推荐卡、OpenClaw 双栏配置区。

原因：

- 逐像素还原最终仍要落在工程可维护的 token 和页面配方上，否则一轮 UI 调整就会牵动大量页面。
- 现有前端规范已经要求样式通过 CSS Variables + SCSS 组织，这与本方案一致。

替代方案：

- 直接在各页面里写死样式。
  - 放弃原因：短期快，但会再次得到一个不可维护的“截图拼接版”。

### 4. 执行台采用“高保真查看器”边界，不在本轮做自由编排器

决定把执行台实现边界定为：可查看节点、可感知运行状态、可切换顶部 tab、可观察日志流、可打开干预面板或干预页、可进行基础缩放/定位控制；不在本轮要求节点拖拽、自由布线和复杂编排编辑。

原因：

- 用户要求的是页面高保真和合理交互，不是重新定义工作流编排产品。
- `_6` 的核心体验是强视觉状态查看和运行干预，而不是自由图编辑器。

替代方案：

- 按设计稿外观一次性做成全功能画布编排器。
  - 放弃原因：超出本轮范围，且会明显延迟交付。

### 5. Mock 运行时需要升级为“视觉驱动 mock”

决定保留现有统一 mock 边界，但把数据层升级为支持：

- 高信息密度卡片内容
- 执行台节点状态序列和日志推送节奏
- 空状态与已初始化态切换
- 缺失素材时的截图占位和文本占位策略

原因：

- 第一版 mock 足以跑通流程，但不足以支撑高保真页面里的状态密度和视觉细节。
- 如果不升级 mock，页面实现最终仍会回退到组件内硬编码视觉假数据。

替代方案：

- 页面各自写临时展示数据。
  - 放弃原因：会破坏前一轮已经建立的 shared mock 边界。

## Risks / Trade-offs

- [Risk] 参考设计稿之间存在不同品牌名、不同导航文案和不同壳层样式。 → Mitigation：统一以 `CyberNomads` 品牌和中文导航落地，只把设计稿当作视觉参考而不是原样拼接。
- [Risk] 高保真要求会显著提升页面样式复杂度。 → Mitigation：先建设共享 token 和通用面板配方，再进入逐页还原，避免重复堆样式。
- [Risk] 某些图标、插画或头像素材缺失，导致页面难以完全复刻。 → Mitigation：允许使用截图切片、占位图形和统一占位资源，并在 design/tasks 中显式记录可替换资源点。
- [Risk] 执行台如果过度追求功能完整，会挤压整体高保真交付节奏。 → Mitigation：明确本轮边界是“高保真查看与干预”，不扩张到全功能编排器。
- [Risk] 顶层 `docs/api` 与 `docs/sql` 为空，真实接入阶段可能暴露字段缺口。 → Mitigation：本次仅定义前端表现和 mock 运行时，不凭空扩展后端契约。

## Migration Plan

1. 先在 OpenSpec 中固定 proposal、design、specs、tasks，明确高保真重建范围。
2. 在实现阶段先重建共享视觉 token、壳层布局、中文导航和品牌资源映射。
3. 其次完成关键参考页还原：工作区列表、工作区创建、执行台、Agent 空状态、OpenClaw 配置页。
4. 再把同一视觉规则扩展到资产、策略、账号等编辑/列表页面。
5. 最后统一校准桌面断点、占位资源、状态反馈和视觉细节偏差。

本次变更面向本地前端工程，不涉及线上迁移。若高保真实现阶段需要替换已有页面骨架，按页面模块逐步替换即可，不需要额外数据迁移。
